### Training-loop initiative (2026-05-09)

The 2026-05-09 session surfaced the central calibration finding: small/local models need much more handholding than Opus-class — abstract style descriptors yield bullets when the project style is prose, while verbatim-example anchoring + explicit anti-hallucination guards turn that same task into a verbatim-usable HIT. The `delegate-feedback.sh hit|miss` infrastructure (#70) tracks calibration empirically; what was missing was the path from "we learned something" to "future sessions inherit it". All four layers are now shipped — ordered cheapest-first — plus a Layer 5 runtime nudge (Phase 8 / #91) that operationalises Layer 4 by surfacing recurring MISS patterns at the moment of insight:

1. **Layer 1 — `prompts/` library + SKILL.md Recipes section.** *Shipped via #72.* One markdown file per recurring task type. Each ships the proven prompt skeleton, the canonical anchors, and the explicit anti-hallucination guards drawn from real session HITs. SKILL.md `## Recipes` section points the agent at the directory; `tests/test-prompts-library.sh` enforces structural validity. Initial recipes: `prompts/commit-message.md`, `prompts/pr-description.md`. Discipline going forward: every recurring HIT graduates to a recipe; every MISS that names a missing recipe gets one filed.

2. **Layer 2 — `delegate.sh --recipe NAME` flag.** *Shipped.* The wrapper takes `--recipe NAME [--var key=value ...]`, loads `prompts/<NAME>.md`, extracts the first fenced block under `## Prompt template`, substitutes each `{{key}}` placeholder from the matching `--var`, and refuses to send a partly-substituted template (unsubstituted placeholders are an exit-2 error listing the missing keys). The implicit `{{stdin}}` placeholder consumes piped context when present. Recipes gain `## Variables` and `## Invocation` sections documenting each placeholder and a copy-pasteable invocation example; `tests/test-prompts-library.sh` enforces every `{{name}}` in a template is documented and that no legacy `<paste ... here>` markers survive. The metrics line carries a `recipe` field for layer-3 telemetry. `DELEGATE_PROMPTS_DIR` env var overrides the recipe directory for tests.

3. **Layer 3 — MCP `recommend_prompt(task)` tool.** *Shipped.* The `mcp/` server's `recommend_prompt(task, include_examples=True, max_examples=3)` tool matches a task description against the `prompts/*.md` recipe stems by token overlap (with British/US-spelling aliases), parses the recipe's `Prompt template` / `Variables` / `Invocation` sections, and walks `~/.claude/skills/delegate-to-ollama/metrics.jsonl` to attach `hit_count`, `miss_count`, and the most recent local HITs with their feedback `reason`. This turns the hit/miss log from a passive scoreboard into active routing signal — Codex, OpenCode, Cursor, and any other MCP-aware client now inherits the same calibration the Claude Code skill consumes via `delegate.sh --recipe`. Honours `DELEGATE_PROMPTS_DIR` and `DELEGATE_METRICS_FILE` for portability and tests.

4. **Layer 4 — Cross-machine signal via tagged issues.** *Shipped.* The `prompt-pattern` GitHub label and `.github/ISSUE_TEMPLATE/prompt-pattern.md` issue template capture recurring MISS patterns the prompts library doesn't cover (task shape, tier and resolved model, verbatim prompt and output, the prompt that turned the MISS into a HIT when known). Issues are the public review surface; a maintainer (or a future PR-bot) graduates an issue into a `prompts/<new>.md` entry, paired with an `evals/eval-set.json` positive that asserts the trigger surface still fires on the task shape. `prompts/README.md` "Cross-machine signal" section documents the flow. Closes the loop empirically — every recurring miss becomes a test case and a fix instead of evaporating after one conversation.

The deeper insight underneath all four layers: small/local models default to whatever shape is most common in their training (bullets when asked for "concise commit message"); they don't fill in the user's stylistic gaps the way larger models do. The training discipline is to write prompts as if for a junior who needs every detail spelled out, then capture what worked.

### Other open priorities

#### Recipe library expansion (2026-05-18)

The 2026-05-18 session re-framed where the recipe library has room to grow. The current library is mostly output-bounded (commit messages, PR descriptions, review replies, release notes) — small inputs, small outputs anchored by templates. The bigger per-call token win is the *input-digestion* shape: ingest a 10-500 KB blob, return ~500 bytes of structured fact. Per-call savings on this shape dominate the output-bounded recipes by 10-100×, which is the case for expanding the library in that direction. `prompts/ci-log-triage.md` shipped in this session as the prototypical input-digestion example; the three remaining candidates below cover both shapes. Each is deferred until its trigger condition is met.

1. **P1 — `prompts/roadmap-entry.md`** (issue [#125](https://github.com/IsmaelMartinez/delegate-to-ollama/issues/125)). Drafting a single "shipped" entry for a long-running project plan / roadmap document, given a structured fact list (PR numbers, squash hashes, dates, per-PR shipped summaries, optional Up-Next pointer) and a verbatim style anchor (an existing entry in the same file). Output-bounded recipe (~500 bytes input fact list + ~2 KB style anchor → 1-2 prose paragraphs). The issue author has used the pattern three times in two days across `bonnie-wee-plot` PRs (#372, #382, #383) and documented the recurring drifts: prose tier defaults to numbered lists when input phrasing mentions "three load-bearing decisions" even with a prose anchor, quiet Americanisation when British spelling is in the anchor, the documented trailing-padding sentence (re-asserting in this recipe because roadmap entries are factual logs where closing "ensuring…" sentences read especially wrong), and occasional PR-number / commit-hash typos. The issue body contains a near-complete recipe draft — the work is translating it into `prompts/roadmap-entry.md`, registering it in `prompts/README.md`, adding an `evals/eval-set.json` paraphrase positive, and dogfooding against one of the in-flight `bonnie-wee-plot` sprints. Trigger to ship: anytime the next session has bandwidth — this one is unusual in the priority list because the issue author has done the calibration work upfront, so the work is mostly mechanical conversion rather than empirical discovery.

2. **P2 — `prompts/long-thread-distillation.md`.** Input: a full PR or issue comment thread (10-50 comments, often 20-100 KB). Output: outstanding action items + who's blocked + current consensus, as three short structured sections. Distinct from `summarise-issue.md` (timeline format) because the use case is "I'm picking up a stale PR and want to know where it stands," not "I'm summarising for a status update." Trigger to ship: any future session where the agent reads a long thread top-to-bottom to triage current state. The most likely trigger surface is `/address-pr-comments` on a thread with many prior rounds of review.

3. **P3 — `prompts/bulk-file-summary.md`.** Input: a list of N files (5-50 typical, 1-20 KB each). Output: one-line description per file. Distinct from `file-summary.md` which is single-file. The amortisation win is the loop — the agent calls the recipe once per file in sequence, and each call is independently small enough to fit the prose-tier ceiling per issue #110. Useful when starting work in a new repo or area of an existing repo. Trigger to ship: any future session where the agent reads ≥ 5 files just to orient before editing.

#### Capability expansion — modality tiers (2026-05-19)

Two of the three modalities (embedding, vision) are scaffold-only in Phase 4 (`pick-model.sh` resolution works for both); audio is not yet scaffolded. None of the three integrate with `delegate.sh` today because the wire-shape differs from the text-in / text-out prose path. The wrapper integration is the next blocker for each; priority is cheapest-implementation first.

1. **P1 — `embedding` tier wire-up.** `nomic-embed-text` resolution shipped in Phase 4 but the call shape is fundamentally different: Ollama exposes embeddings via `POST /api/embed` (no `ollama` CLI subcommand) and the response is a vector, not text. The work is a dedicated `scripts/embed.sh` (the metrics + recipe system assumes a text output, so reusing `delegate.sh` would force several special cases) plus one or two recipes that wrap it usefully. `prompts/semantic-search.md` ("given a query and a list of N files, return the top-K by cosine similarity") is the obvious first recipe and unlocks the "find the runbook that mentions X" pattern the agent currently solves by reading every file. Cheap to implement (no new model pull, cosine similarity over normalised embeddings reduces to a dot product in numpy or jq) and the per-call calling-side saving is high — every file the agent skips is 1-20 KB of Sonnet input avoided. Trigger to ship: next session with bandwidth.

2. **P2 — `vision` tier wire-up.** Routing landed in Phase 4; resolution gated on `ollama pull qwen3-vl:30b-a3b-thinking` (or a vision-capable Qwen3.6). Wire-shape work is base64-encoding into the `images: [...]` array on `POST /api/generate` — `ollama` CLI 0.21 has no `--image` flag so this also bypasses `delegate.sh` until the wrapper learns an image path. Primary use cases: screenshot OCR, diagram-to-prose, photo description. The exploratory "draw me a diagram (mermaid/ascii)" pattern surfaced in the 2026-05-19 framing — the calling agent feeds a scope description and the local model emits a diagram syntax for the main agent to consume as a scope-understanding check — is text-out not vision-in, so it shares the wrapper hook only loosely and does not strictly need the `vision` tier. Defer until either an OCR / screenshot trigger condition shows up in real sessions or the diagram-generation hypothesis gets a deliberate probe.

3. **P3 — Audio tier (STT).** Not yet scaffolded — would need a new `audio` tier in `pick-model.sh` plus a separate binary (Ollama doesn't ship a speech model; `whisper.cpp` or `mlx-whisper` is the obvious local path). Use case is "transcribe this meeting recording / voice memo, then summarise" — pairs naturally with the existing prose recipes once the transcript exists. Wrapper integration is more involved because the input is a file path rather than piped text. The most speculative of the three; defer until a concrete trigger.

## Phase 9 — First-run personalisation

The skill ships one set of `pick-model.sh` preferences and assumes the user's installed model set roughly aligns with the shipped order. In practice every machine has a different mix; the skill should adapt itself to the host without forcing a repo fork. Prior art (aider's `~/.aider.model.settings.yml` layered config, Continue.dev's `model: AUTODETECT`, SmarterRouter's `setup` wizard, Claude Code's `.claude/settings.local.json` per-user override) all point at the same shape: ship a community default, let users drop a personal override on top, keep the override outside the repo so `git clean` can't eat it.

- [done] **v1: per-user override hook + read-only init script.** `pick-model.sh` sources `${DELEGATE_TO_OLLAMA_CONFIG:-$HOME/.claude/skills/delegate-to-ollama/config.sh}` after the shipped defaults populate `prefs`; the override is plain bash that may reassign `prefs` per tier. Untouched tiers fall through to shipped defaults; an absent override changes nothing. `scripts/init.sh` reads `ollama list`, parses the shipped prefs from `pick-model.sh`'s case statement (single source of truth), and emits a starter override to stdout with installed-first ordering — never auto-writes. README "Personalising routing" subsection documents the flow. 13 new tests cover override-replaces, override-leaves-others-alone, override-absent-defaults-win, dry-run-trace-surfaces-override, and init.sh round-trip.
- llmfit-driven re-ranking. The v1 init.sh preserves shipped order and only promotes installed models; it doesn't yet consult `llmfit recommend --json` to re-rank by hardware fit-score. Worth adding once a user reports that the shipped order is materially wrong for their hardware.
- Metrics-driven size-aware routing. Today `prose` resolves to the same model (`qwen3.6:35b` on the reference host) for a 40-word commit-message body and a 200-word runbook paragraph alike — the field-notes pass on plg-tech-cloudfront-waf flagged that the smallest tasks burn power without a matching quality requirement. Out of scope until there is evidence the demotion would not hurt quality on a measurable fixture (Phase 7 framework can answer this); listed here so the idea isn't lost. Pairs naturally with the `delegate.sh` `"still waiting at 30 s…"` diagnostic — both come out of the same field-notes pass.
- Empirical micro-benchmark step. ## Phase 10 — Delegation discipline (from 2026-05-03 retrospective)

The 2026-05-03 director-with-worker experiment moved the local model from 1.5/4 to 3.6/4 — within 9% of Haiku 4.5 — by applying four discipline practices: atomic per call, one-shot example, explicit qualifier rules, thinking off. The chain of follow-up probes (v3 through v8 plus a size-floor and an adversarial-test sequence) is logged under `experiments/sessions/2026-05-03-*` and `2026-05-04-*` with per-session `RETROSPECTIVE.md` files. v5 confirmed the directive-rule pattern as a fifth discipline practice; v7 confirmed it is task-shape rather than task-content discipline. v6 found that reasoning architecture beats parameter count and promoted `deepseek-r1:32b` ahead of `phi4-reasoning` in the reasoning tier (rationale captured in `docs/adr/0005-reasoning-tier-ordering.md`); the size-floor probe placed the directive-rule threshold between 9 and 19 GB within the deepseek-r1 family. v8 demonstrated minimal-patch code delegation under SEARCH/REPLACE format at ~250× lower per-cell cost than Opus, and the adversarial chain surfaced the REFUSE-hatch pattern as a sixth discipline-bullet candidate. The chain also disconfirmed two hypotheses: format-schema constrained decoding does not close the calibration gap (v3), and a single counterintuitive example does not move the model's prior (v4).

Skill changes shipped from this chain: `delegate.sh` (`--think=false` default, HTTP-API switch from `ollama run` CLI), `SKILL.md` (Fits / Discipline / tier-routing edits, REFUSE-hatch language, prose/code-contradiction failure mode), `pick-model.sh` (reasoning-tier order with `:32b` substring tightening), and `tests/run-tests.sh` (regression assertion encoding the v6 baseline finding). Upstream contribution to Ollama issue #14645 (format-ignored-when-thinking-disabled) logged on 2026-05-04 with disambiguated negative reproduction.

Still open — **Director-side severity reweighting pattern (fallback).** When calibration is genuinely subjective and shouldn't be hardcoded, the alternative to a directive rule is for the calling agent to accept the model's raw severity and apply qualifier-aware adjustments itself. Document in `SKILL.md` if/when a real consumer reports a calibration disagreement that the directive-rule pattern can't capture.

## Phase 11 — External observability (OpenTelemetry exporter and dashboards)

This phase extends Phase 8's on-disk JSONL telemetry out to an external OpenTelemetry backend so hit-rate and miss-reason dashboards live in a real observability tool rather than a JSONL file queried only by `metrics-summary.sh`. An opt-in OTLP exporter, already filed as issue #134, implements a schema that mixes OTel's `gen_ai.*` semantic conventions for things they cover and a private `delegate.*` namespace for things they don't (verdict, recipe, tokens-avoided), with a default platform recommendation of Grafana Cloud free tier for its pre-built GenAI dashboards plus a privacy-conscious local fallback of Langfuse self-hosted via docker compose. The skill stays "two bash scripts" — `jq` plus `curl`, no Python or Go SDK runtime dependency — and content never leaves the host; only char counts, tier, model, backend, recipe name, and the hit/miss verdict travel to the collector.

The research anchors are three findings from the 2026-05-21 planning sweep. First, OTel's `gen_ai.*` conventions are still in Development status (no attribute or metric is yet Stable) but cover everything the skill cares about per call — `gen_ai.operation.name`, `gen_ai.provider.name` (with `ollama` and `mlx` already registered as provider strings), `gen_ai.request.model`, plus the `gen_ai.client.operation.duration` histogram that Grafana's pre-built GenAI dashboards key off. Second, no `gen_ai.evaluation.*` namespace exists for the hit/miss verdict — Langfuse models user feedback as a first-class `scores` object keyed by `traceId`, and the OTel-native equivalent is to emit the feedback as a new short span in a new trace whose `links` carry the parent's `traceId`/`spanId`, which is the documented pattern for the human-in-the-loop case where feedback can arrive minutes or hours later. Third, the plg-core skill-pack the issue author named as inspiration is private to that organisation; what its public surface reveals is an EventBridge envelope shape, not OTel native — so plg-core is an example of a parallel approach taken elsewhere rather than a baseline to copy.

The work splits into seven independently-shippable tracks designed so most can land in parallel agent sessions:

- **Track A — OTLP exporter MVP** *(issue [#134](https://github.com/IsmaelMartinez/delegate-to-ollama/issues/134) — already filed, narrow scope).* `delegate.sh` posts one span per invocation to `${DELEGATE_OTEL_ENDPOINT}/v1/traces` when the env var is set, otherwise no behaviour change. `delegate-feedback.sh` posts a follow-up feedback span with `links: [{traceId, spanId}]` pointing to the parent delegate span. Trace and span IDs are captured into the existing metrics JSONL row at delegation time (`otel_trace_id`, `otel_span_id` keys) so the feedback exporter can correlate without a second lookup. Env-var surface: `DELEGATE_OTEL_ENDPOINT`, `DELEGATE_OTEL_HEADERS` (pass-through), `DELEGATE_OTEL_SERVICE_NAME` (default `delegate-to-ollama`), `DELEGATE_OTEL_TIMEOUT` (default `1` — kept short because the timeout fires on every call when the collector is unreachable, and telemetry is non-fatal). Failures non-fatal. Tests mock `curl` and assert payload shape, header pass-through, and that a collector timeout does not change the delegation's exit status.
- ✅ **Track B — Schema ADR + reference doc** *(shipped #164)*. ADR under `docs/adr/` documenting the `gen_ai.*` vs `delegate.*` namespace split, the feedback-as-linked-span pattern (rather than parent-span events or trace reuse), and the explicit decision not to emit prompt/output content as attributes. Companion `docs/otel-schema.md` lists every attribute the exporter emits with its source JSONL field and OTel convention reference. No code in this track — it gates Track A's review by writing down the decisions Track A encodes.
- ✅ **Track C — Platform setup runbooks** *(shipped #166)*. `docs/observability/grafana-cloud.md` walking through Grafana Cloud free-tier signup, OTLP endpoint URL, the Basic-auth `instanceID:apiToken` shape, and pointing at the pre-built GenAI dashboard set. `docs/observability/langfuse-self-host.md` with a one-file `docker-compose.yml` that brings up Langfuse plus ClickHouse, the `/api/public/otel/v1/traces` endpoint config, and the API-key extraction step. `docs/observability/phoenix.md` documenting the `docker run -p 6006:6006 arizephoenix/phoenix` one-liner as an ultra-light alternative. Each runbook ends with a copy-pasteable `export DELEGATE_OTEL_ENDPOINT=… ; export DELEGATE_OTEL_HEADERS=…` block.
- **Track D — Dashboards as code.** A `dashboards/grafana-genai.json` Grafana dashboard JSON checked into the repo with panels for hit-rate over time, miss-reason word cloud or top-N list, tokens-avoided counter, per-recipe and per-tier slices, and exit-status-3 (canary-failure) rate. A parallel `dashboards/langfuse-views.json` describing the same views for Langfuse's programmatic API. Both are reproducible-from-scratch artefacts so a new user can land a working dashboard set in one import step.
- **Track E — Historical JSONL backfill.** `scripts/backfill-otel.sh` reads the existing `metrics.jsonl`, generates synthetic trace/span IDs for rows that pre-date the exporter (writing the IDs back to the JSONL row so a partial-progress re-run resumes per-row rather than restarting from scratch), and posts them with their original timestamps so the dashboards have history on day 1 rather than starting empty. Idempotency is row-level: any row that already has `otel_trace_id` is skipped, so an interrupted run can resume without duplicating spans in the backend. Separate PR from Track A because the export-from-history shape differs from the export-on-each-call shape.
- **Track F — Privacy / redaction guard.** Unit-test assertion that no attribute key matching `gen_ai.prompt` / `gen_ai.completion` / `delegate.prompt_text` / `delegate.output_text` is ever present in the OTLP body, regardless of env-var settings. Companion `docs/observability/privacy.md` enumerating exactly what does and does not leave the host. Closes the gap between the README's privacy framing and the actual wire payload.
- **Track G — Metrics counters (deferred behind Track A).** After Track A spans land, decide whether to emit the `gen_ai.client.operation.duration` histogram and a custom `delegate_to_ollama.tokens_avoided` counter via `/v1/metrics`, or whether spans-only is sufficient for the dashboards Track D builds. Default decision: spans-only for v1, metrics in v2. Track G exists so the decision is recorded rather than implicit.

## Phase 12 — Prompt-library hardening (evidence-gated import, lazy migration)

The recipe library has accumulated calibration through real MISS evidence — every guard line in `prompts/<task>.md` is tied to a dated observation with a scorer-aligned re-measurement where the recipe has one. The 2026-05-22 research sweep (sst/opencode and QwenLM/qwen-code internal prompt systems, Microsoft Prompty + fabric + Anthropic prompt library + LangChain hub, plus the persona-and-domain-priming literature for Qwen-class small models) confirmed two things at once. Other coding agents do explicit persona-and-scope openers; the empirical literature (Zheng et al. arXiv 2311.10054, Wharton "Playing Pretend" report, OpenReview "brilliant mathematician") shows persona is null-to-negative on closed-form tasks for this exact model class. The literature flagged domain-context priming as the candidate addition for empirical validation; Track A below was that test.

Phase 12 entered with the literature-derived hypothesis that domain-context priming would benefit small Qwen-class models, yet Track A (#160, shipped as #168) returned a null result across ninety deterministic reps (3 recipes × 3 variants × 2 models × 5 reps). The domain-priming variant failed to beat the control in any of the six (model, recipe) comparisons and regressed on commit-message tasks because the preamble exceeded the character budget. Persona negative-controls matched or underperformed the control, aligning with Zheng et al. arXiv 2311.10054, while standard deviations remained zero due to deterministic Ollama outputs. The data suggests preamble length is the load-bearing variable rather than the semantic category of the priming. Phase 12 therefore ships the rejection-rationale documentation (#167) and the audience-filter port from sst/opencode (#165) unconditionally. The future-recipe convention in Track B (#161) reduces to the identity-and-scope opener and the flat-YAML inputs piece.

The phase is deliberately lighter than the local-brain-equivalent overhaul the user explicitly ruled out. Sub-tracks:

- ✅ **Track A — Domain-priming experiment (validation gate)** *(shipped #168 — **null result**, domain-priming NOT adopted)*. Built three variants of `commit-message.md`, `prompts/file-summary.md`, and `summarise-issue.md`: (1) current, (2) current plus a one-line domain stipulation at the top of `## Prompt template` ("The input is a unified diff hunk plus three recent commits as shape anchors"), (3) current plus a "You are an expert X" persona prefix as a negative-control. Ran against `qwen3.6:35b-a3b-q8_0` and `qwen3-coder:30b-a3b-q8_0` at `--reps 5` and scored with existing T4 plus new T7 (`file-summary`) and T8 (`summarise-issue`) deterministic scorers. The acceptance gate (variant 2 beats variant 1 by ≥0.5 mean score with non-overlapping stdev intervals; variant 3 does not exceed variant 2) was not met on any of the six (model, recipe) comparisons. Full results under `experiments/results/2026-05-22-phase-12-track-a-domain-priming.md`. The two reusable T7/T8 scorers stay in the repo as infrastructure for future recipe-iteration probes that need a similar structural rubric.
- **Track B — Future-recipe convention (lazy migration).** Update `prompts/README.md` "Adding a new recipe" with an optional convention: a one-sentence identity-and-scope opener at the start of `## Prompt template` (NOT persona — task plus forbidden actions, consolidating directives that today are scattered under `## Anti-hallucination guards`) and an optional `inputs:` block in frontmatter for type hints that `delegate.sh --recipe NAME` can validate pre-flight. The `inputs:` block is constrained to flat `key: type` pairs only (no nesting, no anchors, no flow style) so `awk` can parse it without pulling in `yq` and breaking the "two bash scripts" rule — heavier YAML features stay rejected by design rather than imported by accident. New recipes adopt the convention. Existing recipes migrate only when touched for other reasons. No bulk reformat — the calibration cost of re-running every scorer is real, the user explicitly bounded effort, and the literature does not predict a quality uplift large enough to justify it.
- ✅ **Track C — Document rejection rationale** *(shipped #167)*. Add a section to `prompts/README.md` explaining why this library does NOT adopt the "you are an expert X" persona shape (Zheng et al. null result on Qwen-class), Microsoft Prompty's wholesale schema (calibrated for VS Code Copilot tooling this skill doesn't consume), or fabric's explicit-item-count discipline (small-model fabrication trigger; no measured benefit on the T4/T5/T6 fixture set). Closes the gap before a future contributor drifts toward the public idiom unknowingly.
- ✅ **Track D — Port the strongest external example (sst/opencode audience-filter)** *(shipped #165)*. One-recipe port study: extract the audience-filter rule, integrate into `release-note.md`, dogfood on the next release-notes pass, record verdict.

## Out of scope

- Code edits, refactors, or feature implementation. Local models are weak agents and the skill description explicitly rejects these. The local-brain finding stands: "they didn't need Smolagents, they needed `git status | ollama run model`".
- Auto-pulling models without confirmation. Multi-GB downloads stay user-decided; the audit script suggests, never installs.
- A general-purpose router that competes with Claude on routing decisions. This skill picks a model within Ollama; it does not decide whether to call Claude vs Ollama. That decision belongs in the skill description, evaluated by Claude itself.
