### Phase 19 — Grounding-checker "second brain" recipe (2026-05-31 — graduated 2026-06-01, reasoning tier, C6 measured-not-gated)

Phase 19 introduces `ground-check`, a delegate-local recipe that turns a local model into a grounding aide for the MAIN agent rather than a recipe over user content. It takes a numbered list of CLAIMS the agent is about to assert plus an EVIDENCE block and returns one verdict per claim — SUPPORTED with a verbatim quote, CONTRADICTED with a verbatim quote, or NOT-STATED — so the agent can catch itself overreaching before shipping an unsupported assertion. It is a closed-form quote-finder, not a judge: the architectural lever is forcing verbatim quotes and then running a deterministic post-check that verifies each quote is an exact substring of the evidence, downgrading any fabricated-quote verdict to UNVERIFIED. The motivation is empirical — verification is the dominant prompt shape across the usage history (351 prompts say verify/confirm/ensure, 149 check-if, 74 did-we-miss) and main-agent overreach is a recurring real error — and a 2026-05-31 probe on the fast prose model scored 5/5 in 13 seconds, correctly marking an overreach claim NOT-STATED instead of rubber-stamping it.

The full design and build sequence live in `docs/research/2026-05-31-ground-check-plan.md`, produced via an eight-agent ultrathink workflow (five design perspectives, an adversarial and a conventions critique, then synthesis). Key resolved decisions: the recipe routes to the reasoning tier per SKILL.md's "reasoning for inferring-about-prose" rule (with a prose-vs-reasoning A/B at the graduation step), carries no `flaky_on_models` gate because nothing is yet measured to anchor one, and ships a single shared `grep -F` substring helper so the scorer and the runtime wrapper cannot drift. Scope is deliberately narrow and advisory: it never gates a merge or ship, it excludes arithmetic (the models are weak at math) and judgment, and a SUPPORTED verdict certifies only that a quote exists, not that the claim is globally true. Status: the scaffold shipped via PR #251 on 2026-06-01 — the recipe, the `score-t9` scorer, the `task-9` fixture, the runtime wrapper, and the offline tests. The graduation A/B added a relevance/scope contrastive anchor (designed via a workflow); after a scope-directive refinement both tiers converge at 0.9722 with non-overlapping stdev (recall 1.0/1.0, zero fabricated-quote SUPPORTED surviving the post-check), clearing every §6 condition except one. That residual is C6, a borderline qualifier-drop (the evidence says dashboards "render correctly in staging"; the claim drops the qualifier to "render correctly"): it proved unbindable via two contrastive prompt levers and via thinking-on deliberation, and the model's SUPPORTED reading is defensible — forcing NOT-STATED would over-flag real claims that drop an incidental qualifier. Per the 2026-06-01 decision, C6 (f6) is therefore measured-but-not-gated (§6(c) amended), and the recipe graduated to shipped on the reasoning tier (prose ties). A side finding worth keeping: the relevance anchor is tier-dependent — it needs the reasoning tier's chain-of-thought to apply correctly (it regressed prose until a scope directive repaired it), vindicating SKILL.md line 141 for inference-about-prose recipes.

### Phase 18 — Expansion: MLX tier coverage + cross-project adoption (2026-05-27)

Phase 18 bundles the three actionable tracks from a four-reviewer planning session (local model, pragmatic engineer, cost analyst, parallelism advisor) that assessed the skill's real-world usage patterns against the Phase 11 observability data and the `delegate.project` attribution shipped in PR #224. The research document lives at `docs/research/2026-05-27-expansion-plan.md`; literature anchors are arXiv 2502.11021 (MLX inference scaling on Apple Silicon), arXiv 2603.04445 (quantisation-aware routing), LintLLM GLSVLSI 2025 (local-model CI integration), and OWASP ASI-08 (model-supply-chain integrity). Phases C (confidence post-check) and D (test-stub recipe) were reviewed and rejected in the same session: C's output-length proxy fails for summarisation which accounts for 87% of delegation volume, and D has no demand signal with only 1 lifetime code-tier call recorded. Three additional expansion candidates (lint-style diff feedback, structured data extraction, changelog batching) are parked under Other open priorities with named triggers.

**Phase A — MLX tier coverage.** The prose tier is the only tier with an MLX model today; code and reasoning fall back to Ollama at roughly 2x the latency measured in the Phase 11 Track A v2 baseline (PR #115). Phase A downloads `mlx-community/DeepSeek-R1-Distill-Qwen-32B-MLX-8Bit` for the reasoning tier and `lmstudio-community/Qwen3-Coder-30B-A3B-Instruct-MLX-8bit` for the code tier via `huggingface-cli download`, runs T3-T6 baselines against each on the MLX backend (`DELEGATE_BACKEND=mlx`), and promotes the models into `pick-model.sh` preferences if scores match the incumbent Ollama models on the same fixtures. The acceptance gate is the same as Phase 7's convention: the MLX candidate must meet or exceed the Ollama incumbent's mean score on every deterministic scorer (T3 citation rate, T4/T5/T6 structural pass rate) with non-overlapping stdev intervals on 3+ reps. If a candidate fails one scorer, it stays out of `pick-model.sh` and the failure is recorded in a dated baseline under `experiments/results/` for the next model-change audit to reference.

**Phase B — Cross-project adoption.** The `delegate.project` attribute shipped in PR #224 (commit `0d6da0b`) tags every metrics row with the calling project's directory name. Early data shows 97% of delegation volume originates from the skill's own repo, which is expected for the development period but indicates a cold-start gap for other projects. Phase B investigates the three candidate bottlenecks identified in the multi-reviewer session: the trigger description may not fire reliably when the skill is loaded into a project whose task shapes differ from the eval-set's positives (testable by running `eval-skill-triggers.sh --ollama` with project-specific paraphrases appended to the eval set), the session context loading may not surface the skill early enough in non-delegate repos (testable by inspecting `SKILL.md` load order in a fresh `claude` session against a sibling project), or user habit may default to inline Claude responses for tasks the skill could handle (testable only by instrumented dogfooding across the portfolio). The deliverable is a diagnostic report under `docs/research/` with per-bottleneck measurements and, if the trigger description is the discriminator, a targeted `evals/eval-set.json` expansion plus a re-run of the `--ollama` gate to confirm recall holds.

**Phase E — Untracked verdict reduction (deferred behind B).** The Phase 8 verdict nudge (`DELEGATE_TO_OLLAMA_NO_VERDICT_NUDGE`) brought the untracked rate from 65% to 46%, but nearly half of delegations still carry no HIT/MISS feedback, which limits the recipe library's calibration signal to barely more than half of real-session evidence. Phase E targets under 20% untracked via a session-end sweep mechanism rather than an hourly nag or a blocking gate — the sweep would scan the metrics JSONL for recent rows without a corresponding feedback row and present a batch prompt at session close, letting the user verdict multiple delegations in one pass. The design is deferred behind Phase B because cross-project adoption will change the volume distribution, and the sweep's UX should be calibrated against the steady-state mix rather than the current single-repo skew.

### Phase 16 — Recipe-iteration round 4: tier-gate + verb-substitution treadmill (2026-05-24)

Phase 16 bundles two recipe-iteration tracks selected from the 22-day trend report following Phase 15: a `flaky_on_models:` tier-gate operationalising `prompts/pr-description.md`'s documented mitigation (Track A) and a scorer + recipe enumeration extension that empirically confirms the verb-substitution treadmill predicted in the Phase 15 calibration note (Track B). Both small. Both validated empirically before this PR opened. Track C is the recipe-authoring convention covering the Track A pattern so future recipes can adopt it.

**Track A — pr-description tier-gate.** The 45% hit rate on `pr-description` (5 HIT / 6 MISS across 11 verdicts) was the standout weak recipe in the Phase 15 trend report. The documented mitigation since 2026-05-10 has been "hand-write the PR description"; the wiring was deferred under "not urgent". Phase 16 ships the wiring as a generic recipe convention: `scripts/delegate.sh` parses an optional frontmatter `flaky_on_models:` list (case-insensitive substrings) after model resolution and refuses with exit 4 when any pattern matches the resolved model. The refusal sits BEFORE the pre-flight canary because the gate is structural ("this recipe won't work reliably on this model class") rather than dynamic ("the model isn't responding right now"). The recipe's frontmatter is grounded in the 2026-05-10/11/12/13 stall measurements — `qwen3.6:35b` and `qwen3-next:80b` plus the underscore/dash naming variants. Empirical verification on the reference host: prose tier (qwen3.6:35b-a3b-q8_0) refused with exit 4 + clear stderr; long-context tier (qwen3-next:80b-a3b-instruct-q8_0) refused with exit 4; code tier (qwen3-coder-next:latest) passes the gate and dispatches normally. The override `DELEGATE_FORCE_FLAKY=1` sends the request anyway, useful for capturing fresh evidence the flaky-class behaviour has changed across model upgrades. 21 new test assertions in `tests/test-delegate.sh` (467 → 477) cover the match/non-match/back-compat/override/case-insensitive paths.

**Track B — PADDING_REGEXES + recipe enumeration extension.** Phase 15 Track A caveat: the model substituted unenumerated participial verbs (`, replacing assertion with data`) when the enumerated ones were anchored against. Phase 16 Track B operationalises the predicted next-iteration: extend `experiments/score-t4.sh` `PADDING_REGEXES` with `replacing`, `supporting`, `reflecting`, `keeping`, `exemplified`, and `This provides`, plus extend `prompts/commit-message.md`'s directive enumeration line to match (Phase 13 scorer-recipe parity convention). Measurement on the same fixture anchors confirms the empirical treadmill: the Phase 15 fixture state scored 18/18 against the prior scorer and 15/18 against the extended scorer (catches `, replacing` + `This provides` on stable input) — scorer fidelity improved. But the Phase 16 fixture state scored 18/18 against the extended scorer because the model shifted to `, moving the calibration from X to Y` and `, including subject length, type prefix, ...` — two participial-tail shapes whose verbs (`moving`, `including`) are NOT in the extended regex set. Per-verb enumeration extension is a treadmill, not a fix.

**Track C — `prompts/README.md` Convention 4** documents the `flaky_on_models:` frontmatter pattern as a recipe-authoring convention with two rules: anchor every listed substring in measured evidence, and list substrings (not exact model names) so the gate adapts across quantisation suffixes and naming-convention drift.

Phase 16 shipped via PR `#209` (squash `d03c14f`, merged 2026-05-24). The generalised `, [a-z]{3,}ing` structural matcher shipped in Phase 17 Track B (PR `#213`, squash `c73e1e9`).

### Phase 15 — Contrastive-anchor sharpening past the directive-binding ceiling (2026-05-24)

Phase 13 closed with the empirical observation that directive-text enumeration on `prompts/commit-message.md` had reached a saturation point under greedy decoding on `qwen3.6:35b-a3b-q8_0` — adding more forbidden-phrase examples to the enumeration was no longer the right lever, and the next iteration was named as either a Wrong/Correct contrastive sharpening or a different structural lever. Phase 15 Track A measures the contrastive-anchor lever empirically against the T4 fixture.

Baseline against the prior `prompts/commit-message.md` template (substituted into the 2026-05-21 anchors as `task-4-commit-message-2026-05-24-pre-phase-15.txt`, 3 reps, prose tier) confirmed the ceiling: 12/18 cumulative, mean 0.67, with every rep failing both SUBJECT_LEN (subject 77 chars) and BODY_NO_PADDING (the model emitted `, providing X` and `, allowing X` participial tails despite both verbs being literally named in the directive enumeration). The intervention expands a single Wrong/Correct anchor for closes-the-gap into four pairs covering the participial-`, providing` shape, the participial-`, allowing` shape, the declarative-`This ensures` shape, and the original closes-the-gap form. The Wrong/Correct content uses domain-neutral subject matter (endpoint validation, migration scripts, rate limiters, contract-vs-payload) rather than recipe-adjacent phrasing, after a first iteration with recipe-adjacent examples scored 18/18 but leaked the "Correct" sentence verbatim into the model's output. Post-edit re-measurement against `task-4-commit-message-2026-05-24.txt` (the new runner default): 18/18 cumulative, mean 1.00, both SUBJECT_LEN (50-char subject) and BODY_NO_PADDING now pass.

Track A's caveat worth carrying forward: one of the 3 reps emitted `, replacing assertion with data` — a participial-tail shape whose verb (`replacing`) is not in the scorer's `PADDING_REGEXES` enumeration. The model now AVOIDS the enumerated verbs but defaults to a structurally-equivalent unenumerated verb. Verb-level enumeration has succeeded on coverage, but the underlying participial-tail STRUCTURE persists. Two future levers are candidates: extending the scorer's regex set with the next batch of unenumerated verbs (`replacing`, `supporting`, `reflecting`, `keeping`) and re-measuring, or attempting a generalised `, [a-z]+ing` trailing-clause matcher with calibrated false-positive thresholds.

Phase 15 Track B measured the same post-edit fixture against the code tier (`qwen3-coder-next:latest`, 3 reps) as a tier-escalation control. All 3 reps appended `(#86)` to the subject despite the explicit Wrong/Correct anchor in the template body, scoring 15/18 cumulative mean 0.83. The code tier scored WORSE than the prose tier on the same recipe, with the `(#NN)`-suffix pattern from the recent-commits anchors being the consistent failure mode. Speculative reading: code-specialised models treat the recent-commits anchors as a sequence to extend (including the trailing `(#NN)` from squash-merges), while prose-specialised models are less prone to this exact-pattern-extension behaviour. The recipe's prose-tier recommendation in the Invocation section stands; code tier is not a viable fallback for commit-message work. The tier-vs-recipe interaction is rule-specific rather than uniformly better or worse — code tier's body paragraphs were clean of participial tails on all 3 reps.

Track C documents the directive-binding-ceiling pattern as a recipe-authoring convention in `prompts/README.md`. When enumeration alone reaches saturation under greedy decoding on small/local models, the three next-levers in priority order are: (1) contrastive Wrong/Correct anchors with domain-neutral content, (2) call-site reinforcement appended to the trailing prompt, (3) structural pattern matching in the scorer. Verb-level enumeration extension is the cheapest first move but caps out when the model substitutes an unenumerated structural equivalent. The convention codifies what `commit-message.md`'s 2026-05-10/12/22/24 calibration entries arrived at through three rounds of MISS evidence.

Phase 15 shipped via PR `#208` (squash `81c3d68`, merged 2026-05-24). The follow-up scorer regex extension shipped in Phase 16 Track B; pr-description tier-gating shipped in Phase 16 Track A.

### Training-loop initiative (2026-05-09)

The 2026-05-09 session surfaced the central calibration finding: small/local models need much more handholding than Opus-class — abstract style descriptors yield bullets when the project style is prose, while verbatim-example anchoring + explicit anti-hallucination guards turn that same task into a verbatim-usable HIT. The `delegate-feedback.sh hit|miss` infrastructure (#70) tracks calibration empirically; what was missing was the path from "we learned something" to "future sessions inherit it". All four layers are now shipped — ordered cheapest-first — plus a Layer 5 runtime nudge (Phase 8 / #91) that operationalises Layer 4 by surfacing recurring MISS patterns at the moment of insight:

1. **Layer 1 — `prompts/` library + SKILL.md Recipes section.** *Shipped via #72.* One markdown file per recurring task type. Each ships the proven prompt skeleton, the canonical anchors, and the explicit anti-hallucination guards drawn from real session HITs. SKILL.md `## Recipes` section points the agent at the directory; `tests/test-prompts-library.sh` enforces structural validity. Initial recipes: `prompts/commit-message.md`, `prompts/pr-description.md`. Discipline going forward: every recurring HIT graduates to a recipe; every MISS that names a missing recipe gets one filed.

2. **Layer 2 — `delegate.sh --recipe NAME` flag.** *Shipped.* The wrapper takes `--recipe NAME [--var key=value ...]`, loads `prompts/<NAME>.md`, extracts the first fenced block under `## Prompt template`, substitutes each `{{key}}` placeholder from the matching `--var`, and refuses to send a partly-substituted template (unsubstituted placeholders are an exit-2 error listing the missing keys). The implicit `{{stdin}}` placeholder consumes piped context when present. Recipes gain `## Variables` and `## Invocation` sections documenting each placeholder and a copy-pasteable invocation example; `tests/test-prompts-library.sh` enforces every `{{name}}` in a template is documented and that no legacy `<paste ... here>` markers survive. The metrics line carries a `recipe` field for layer-3 telemetry. `DELEGATE_PROMPTS_DIR` env var overrides the recipe directory for tests.

3. **Layer 3 — MCP `recommend_prompt(task)` tool.** *Shipped.* The `mcp/` server's `recommend_prompt(task, include_examples=True, max_examples=3)` tool matches a task description against the `prompts/*.md` recipe stems by token overlap (with British/US-spelling aliases), parses the recipe's `Prompt template` / `Variables` / `Invocation` sections, and walks `~/.claude/skills/delegate-local/metrics.jsonl` to attach `hit_count`, `miss_count`, and the most recent local HITs with their feedback `reason`. This turns the hit/miss log from a passive scoreboard into active routing signal — Codex, OpenCode, Cursor, and any other MCP-aware client now inherits the same calibration the Claude Code skill consumes via `delegate.sh --recipe`. Honours `DELEGATE_PROMPTS_DIR` and `DELEGATE_METRICS_FILE` for portability and tests.

4. **Layer 4 — Cross-machine signal via tagged issues.** *Shipped.* The `prompt-pattern` GitHub label and `.github/ISSUE_TEMPLATE/prompt-pattern.md` issue template capture recurring MISS patterns the prompts library doesn't cover (task shape, tier and resolved model, verbatim prompt and output, the prompt that turned the MISS into a HIT when known). Issues are the public review surface; a maintainer (or a future PR-bot) graduates an issue into a `prompts/<new>.md` entry, paired with an `evals/eval-set.json` positive that asserts the trigger surface still fires on the task shape. `prompts/README.md` "Cross-machine signal" section documents the flow. Closes the loop empirically — every recurring miss becomes a test case and a fix instead of evaporating after one conversation.

The deeper insight underneath all four layers: small/local models default to whatever shape is most common in their training (bullets when asked for "concise commit message"); they don't fill in the user's stylistic gaps the way larger models do. The training discipline is to write prompts as if for a junior who needs every detail spelled out, then capture what worked.

#### Strategic-review topics to explore (2026-06-04)

A 2026-06-04 strategic review (plan-vs-impasse, stress-testing, model reusability, model currency, transferability) surfaced six explorable topics. They are batched here rather than promoted to the active priority list because most are investigations with a named first action, not committed builds. Framing context: the recipe-calibration lever is at its ceiling (the directive-binding and padding findings of Phases 15–17), MLX tier coverage (Phase 18-A) is now satisfied by the 2026-06-03 baseline (#255) which measured all three routed MLX tiers, and the remaining headroom is in new use-shapes, robustness breadth, and adoption rather than further prompt-tuning.

**A — Reasoning-tier model currency, hardware-gated via `llmfit`.** *Corrected 2026-06-05 after the DeepSeek V4 attempt — the original framing ignored the 128 GB envelope, which was the mistake.* The reasoning tier routes to `deepseek-r1:32b`, an R1 *distill*. The original plan was to audit DeepSeek V4 (Flash/Pro) as the reasoning upgrade, but V4 does not fit this host: V4-Pro (1.6T, ~865 GB weights) needs 512 GB+ even at Q4, V4-Flash (284B, ~160 GB weights) only fits 128 GB at 2-bit (Q2_K 103 GB / MLX `2bit-DQ`) while Q3 (136 GB) and Q4 (142–172 GB) are over budget, and no 32B-class V4 distill — the apples-to-apples comparator — exists yet. So the reasoning-grade V4 is unrunnable locally and a 2-bit speed variant is not a fair reasoning test. The corrected method is to gate every model-currency move through `scripts/audit-models.sh` FIRST, which runs `llmfit recommend --min-fit good` (hardware-fit-filtered, first-party): it never surfaces V4 because V4 does not fit, which is exactly the check that should have run before the V4 detour. The 2026-06-05 run reports no strong upgrade overall (installed models lead or match for this hardware), with one open reasoning question: llmfit ranks `Qwen/Qwen3-Next-80B-A3B-Thinking` (97.9, ~84 GB — the thinking variant of the already-installed 80B instruct) and `openai/gpt-oss-120b` (97.2, fits at 4-bit) above the installed models, while the reasoning incumbent `deepseek-r1:32b` is absent from llmfit's top five. First action: pull `Qwen3-Next-80B-A3B-Thinking`, run `scripts/model-change-audit.sh <model> reasoning` plus a T4–T6 baseline against the `deepseek-r1:32b` incumbent, and adopt only on the Phase 7 non-overlapping-stdev gate; optionally also audit `gpt-oss-120b`. Expectation management: this is currency, not a fix for the highest-volume gap (see topic F), and `scripts/audit-models.sh` is the mandatory hardware-fit pre-filter for any future tier-model change — never assume a frontier model is usable on size alone.

**B — Cross-model recipe-robustness probe.** Every recipe is tuned against Qwen3/DeepSeek failure modes (anti-padding enumeration, contrastive anchors, greedy decoding), and SKILL.md is dense with model-specific findings — so it is unknown whether a given recipe score reflects a robust instruction or a Qwen3-overfit prompt. First action: run the existing T4/T5/T6/T9 scorers against at least one *non-routed* model family (e.g. a Gemma 4 or a Llama-family local model) without re-tuning; a score collapse localises overfit, a holding score evidences a portable instruction. Feeds the reusability question and calibrates trust in the `scripts/model-change-audit.sh` gate. Trigger: do this before adopting any new model family, and opportunistically whenever a non-Qwen model is already installed.

**C — Fixture-distribution hardening.** Each deterministic scorer (T3–T9) rides a *single* fixture at 3 reps — strong for regression, blind to distribution, so a 1.00 may be single-fixture overfit rather than a real gain. First action: add 2–3 varied fixtures per high-traffic scorer (different domains, lengths, and deliberately padding-baiting / fabrication-baiting inputs) and report per-fixture variance so genuine recipe gains separate from fixture-specific luck. Cheapest-first: start with T4 (commit-message, highest volume) and T9 (ground-check, youngest and least exercised).

**D — Ground-check adversarial stress battery.** `task-9` is a 12-claim proof-of-concept over one evidence block; the recipe's "SUPPORTED = a verbatim quote exists" contract has a known seam (the C6 qualifier-drop, measured-but-not-gated). First action: extend the fixture with paraphrased claims, multi-hop inferences, numeric claims (which the recipe is designed to refuse), and longer evidence, to map where the quote-finder's SUPPORTED reading diverges from truth and whether C6-style qualifier-drops generalise into a gateable pattern or stay irreducibly borderline. Keeps the advisory-only scope; the deliverable is a measurement, not necessarily a recipe change.

**E — Reusable-harness vs personal-calibration split (feeds Phase 18-B).** The skill ships as a transferable harness (routing, tiers, scorers, audit gate, and universal recipes like commit-message / summarise-issue / file-summary) plus a layer of defaults calibrated to one person's prose taste (anti-padding, anti-persona, prose-over-bullets — the maintainer's own working preferences compiled into prompts). The 2026-05-27 cross-project-adoption doc attributes low uptake to other repos being code-oriented and lacking delegation hints; this topic adds the second bottleneck — taste-calibrated recipes read as mis-calibrated to a user with different preferences. First action: classify each recipe as universal vs taste-calibrated in `prompts/README.md`, and sketch a "calibration reset" path (how a new adopter re-runs the hit/miss loop against their own taste) so adoption inherits the harness without inheriting one person's defaults. Directly serves the Phase 18-B diagnostic.

**F — Accept or re-lever the model-independent padding ceiling.** The 2026-06-03 baseline proved the T4 `BODY_NO_PADDING` failure is model-independent (all three routed MLX models failed identically, zero variance), and prior rounds showed it unbindable by prompt directives or thinking-mode. The decision this topic forces: either formally accept the ceiling and stop iterating the commit-message recipe's prose lever (it is the highest-volume recipe and the most-iterated), or move the lever off the prompt entirely — e.g. a deterministic post-generation padding strip in the wrapper, reusing the `PADDING_REGEXES` the scorer already encodes as a cleanup pass, which sidesteps the model's prose habit rather than fighting it. First action: a spike on wrapper-side post-strip feasibility (does removing a padding tail ever corrupt a legitimate body?) weighed against a written "accepted ceiling" note that retires further prompt rounds.

## Phase 9 — First-run personalisation

The skill ships one set of `pick-model.sh` preferences and assumes the user's installed model set roughly aligns with the shipped order. In practice every machine has a different mix; the skill should adapt itself to the host without forcing a repo fork. Prior art (aider's `~/.aider.model.settings.yml` layered config, Continue.dev's `model: AUTODETECT`, SmarterRouter's `setup` wizard, Claude Code's `.claude/settings.local.json` per-user override) all point at the same shape: ship a community default, let users drop a personal override on top, keep the override outside the repo so `git clean` can't eat it.

- [done] **v1: per-user override hook + read-only init script.** `pick-model.sh` sources `${DELEGATE_TO_OLLAMA_CONFIG:-$HOME/.claude/skills/delegate-local/config.sh}` after the shipped defaults populate `prefs`; the override is plain bash that may reassign `prefs` per tier. Untouched tiers fall through to shipped defaults; an absent override changes nothing. `scripts/init.sh` reads `ollama list`, parses the shipped prefs from `pick-model.sh`'s case statement (single source of truth), and emits a starter override to stdout with installed-first ordering — never auto-writes. README "Personalising routing" subsection documents the flow. 13 new tests cover override-replaces, override-leaves-others-alone, override-absent-defaults-win, dry-run-trace-surfaces-override, and init.sh round-trip.
- llmfit-driven re-ranking. The v1 init.sh preserves shipped order and only promotes installed models; it doesn't yet consult `llmfit recommend --json` to re-rank by hardware fit-score. Worth adding once a user reports that the shipped order is materially wrong for their hardware.
- Metrics-driven size-aware routing. Today `prose` resolves to the same model (`qwen3.6:35b` on the reference host) for a 40-word commit-message body and a 200-word runbook paragraph alike — the field-notes pass on plg-tech-cloudfront-waf flagged that the smallest tasks burn power without a matching quality requirement. The metrics JSONL already records `prompt_chars`, `output_chars`, `duration_ms`, and `exit_status` per call; a small post-processor inside `pick-model.sh` could demote calls below an input/output size threshold to the next-smaller candidate within the same tier when the rolling success rate is at 100%. Out of scope until there is evidence the demotion would not hurt quality on a measurable fixture (Phase 7 framework can answer this); listed here so the idea isn't lost. - Empirical micro-benchmark step. ## Phase 10 — Delegation discipline (from 2026-05-03 retrospective)

The 2026-05-03 director-with-worker experiment moved the local model from 1.5/4 to 3.6/4 — within 9% of Haiku 4.5 — by applying four discipline practices: atomic per call, one-shot example, explicit qualifier rules, thinking off. The chain of follow-up probes (v3 through v8 plus a size-floor and an adversarial-test sequence) is logged under `experiments/sessions/2026-05-03-*` and `2026-05-04-*` with per-session `RETROSPECTIVE.md` files. v5 confirmed the directive-rule pattern as a fifth discipline practice; v7 confirmed it is task-shape rather than task-content discipline. v6 found that reasoning architecture beats parameter count and promoted `deepseek-r1:32b` ahead of `phi4-reasoning` in the reasoning tier (rationale captured in `docs/adr/0005-reasoning-tier-ordering.md`); the size-floor probe placed the directive-rule threshold between 9 and 19 GB within the deepseek-r1 family. v8 demonstrated minimal-patch code delegation under SEARCH/REPLACE format at ~250× lower per-cell cost than Opus, and the adversarial chain surfaced the REFUSE-hatch pattern as a sixth discipline-bullet candidate. The chain also disconfirmed two hypotheses: format-schema constrained decoding does not close the calibration gap (v3), and a single counterintuitive example does not move the model's prior (v4).

Skill changes shipped from this chain: `delegate.sh` (`--think=false` default, HTTP-API switch from `ollama run` CLI), `SKILL.md` (Fits / Discipline / tier-routing edits, REFUSE-hatch language, prose/code-contradiction failure mode), `pick-model.sh` (reasoning-tier order with `:32b` substring tightening), and `tests/run-tests.sh` (regression assertion encoding the v6 baseline finding). Upstream contribution to Ollama issue #14645 (format-ignored-when-thinking-disabled) logged on 2026-05-04 with disambiguated negative reproduction.

Still open — **Director-side severity reweighting pattern (fallback).** When calibration is genuinely subjective and shouldn't be hardcoded, the alternative to a directive rule is for the calling agent to accept the model's raw severity and apply qualifier-aware adjustments itself. Document in `SKILL.md` if/when a real consumer reports a calibration disagreement that the directive-rule pattern can't capture.

## Phase 11 — External observability (OpenTelemetry exporter and dashboards)

This phase extends Phase 8's on-disk JSONL telemetry out to an external OpenTelemetry backend so hit-rate and miss-reason dashboards live in a real observability tool rather than a JSONL file queried only by `metrics-summary.sh`. An opt-in OTLP exporter, already filed as issue #134, implements a schema that mixes OTel's `gen_ai.*` semantic conventions for things they cover and a private `delegate.*` namespace for things they don't (verdict, recipe, tokens-avoided), with a default platform recommendation of Grafana Cloud free tier for its pre-built GenAI dashboards plus a privacy-conscious local fallback of Langfuse self-hosted via docker compose. The skill stays "two bash scripts" — `jq` plus `curl`, no Python or Go SDK runtime dependency — and content never leaves the host; only char counts, tier, model, backend, recipe name, and the hit/miss verdict travel to the collector.

The research anchors are three findings from the 2026-05-21 planning sweep. First, OTel's `gen_ai.*` conventions are still in Development status (no attribute or metric is yet Stable) but cover everything the skill cares about per call — `gen_ai.operation.name`, `gen_ai.provider.name` (with `ollama` and `mlx` already registered as provider strings), `gen_ai.request.model`, plus the `gen_ai.client.operation.duration` histogram that Grafana's pre-built GenAI dashboards key off. Second, no `gen_ai.evaluation.*` namespace exists for the hit/miss verdict — Langfuse models user feedback as a first-class `scores` object keyed by `traceId`, and the OTel-native equivalent is to emit the feedback as a new short span in a new trace whose `links` carry the parent's `traceId`/`spanId`, which is the documented pattern for the human-in-the-loop case where feedback can arrive minutes or hours later. Third, the plg-core skill-pack the issue author named as inspiration is private to that organisation; what its public surface reveals is an EventBridge envelope shape, not OTel native — so plg-core is an example of a parallel approach taken elsewhere rather than a baseline to copy.

The work splits into seven independently-shippable tracks designed so most can land in parallel agent sessions:

- **Track A — OTLP exporter MVP** *(issue [#134](https://github.com/IsmaelMartinez/delegate-local/issues/134) — already filed, narrow scope).* `delegate.sh` posts one span per invocation to `${DELEGATE_OTEL_ENDPOINT}/v1/traces` when the env var is set, otherwise no behaviour change. `delegate-feedback.sh` posts a follow-up feedback span with `links: [{traceId, spanId}]` pointing to the parent delegate span. Trace and span IDs are captured into the existing metrics JSONL row at delegation time (`otel_trace_id`, `otel_span_id` keys) so the feedback exporter can correlate without a second lookup. Env-var surface: `DELEGATE_OTEL_ENDPOINT`, `DELEGATE_OTEL_HEADERS` (pass-through), `DELEGATE_OTEL_SERVICE_NAME` (default `delegate-local`), `DELEGATE_OTEL_TIMEOUT` (default `1` — kept short because the timeout fires on every call when the collector is unreachable, and telemetry is non-fatal). Failures non-fatal. Tests mock `curl` and assert payload shape, header pass-through, and that a collector timeout does not change the delegation's exit status.
- ✅ **Track B — Schema ADR + reference doc** *(shipped #164)*. ADR under `docs/adr/` documenting the `gen_ai.*` vs `delegate.*` namespace split, the feedback-as-linked-span pattern (rather than parent-span events or trace reuse), and the explicit decision not to emit prompt/output content as attributes. Companion `docs/otel-schema.md` lists every attribute the exporter emits with its source JSONL field and OTel convention reference. No code in this track — it gates Track A's review by writing down the decisions Track A encodes.
- ✅ **Track C — Platform setup runbooks** *(shipped #166)*. `docs/observability/grafana-cloud.md` walking through Grafana Cloud free-tier signup, OTLP endpoint URL, the Basic-auth `instanceID:apiToken` shape, and pointing at the pre-built GenAI dashboard set. `docs/observability/langfuse-self-host.md` with a one-file `docker-compose.yml` that brings up Langfuse plus ClickHouse, the `/api/public/otel/v1/traces` endpoint config, and the API-key extraction step. `docs/observability/phoenix.md` documenting the `docker run -p 6006:6006 arizephoenix/phoenix` one-liner as an ultra-light alternative. Each runbook ends with a copy-pasteable `export DELEGATE_OTEL_ENDPOINT=… ; export DELEGATE_OTEL_HEADERS=…` block.
- **Track D — Dashboards as code.** A `dashboards/grafana-genai.json` Grafana dashboard JSON checked into the repo with panels for hit-rate over time, miss-reason word cloud or top-N list, tokens-avoided counter, per-recipe and per-tier slices, and exit-status-3 (canary-failure) rate. A parallel `dashboards/langfuse-views.json` describing the same views for Langfuse's programmatic API. Both are reproducible-from-scratch artefacts so a new user can land a working dashboard set in one import step.
- **Track E — Historical JSONL backfill.** `scripts/backfill-otel.sh` reads the existing `metrics.jsonl`, generates synthetic trace/span IDs for rows that pre-date the exporter (writing the IDs back to the JSONL row so a partial-progress re-run resumes per-row rather than restarting from scratch), and posts them with their original timestamps so the dashboards have history on day 1 rather than starting empty. Idempotency is row-level: any row that already has `otel_trace_id` is skipped, so an interrupted run can resume without duplicating spans in the backend. Separate PR from Track A because the export-from-history shape differs from the export-on-each-call shape.
- **Track F — Privacy / redaction guard.** Unit-test assertion that no attribute key matching `gen_ai.prompt` / `gen_ai.completion` / `delegate.prompt_text` / `delegate.output_text` is ever present in the OTLP body, regardless of env-var settings. Companion `docs/observability/privacy.md` enumerating exactly what does and does not leave the host. Closes the gap between the README's privacy framing and the actual wire payload.
- **Track G — Metrics counters (deferred behind Track A).** After Track A spans land, decide whether to emit the `gen_ai.client.operation.duration` histogram and a custom `delegate_to_ollama.tokens_avoided` counter via `/v1/metrics`, or whether spans-only is sufficient for the dashboards Track D builds. Default decision: spans-only for v1, metrics in v2. Track G exists so the decision is recorded rather than implicit.

Tracks D, E, F can ship in parallel after Track A. Track G is the post-Track-A decision point. Each remaining track is sized for a single PR and a single agent session.

## Phase 12 — Prompt-library hardening (evidence-gated import, lazy migration)

The recipe library has accumulated calibration through real MISS evidence — every guard line in `prompts/<task>.md` is tied to a dated observation with a scorer-aligned re-measurement where the recipe has one. The 2026-05-22 research sweep (sst/opencode and QwenLM/qwen-code internal prompt systems, Microsoft Prompty + fabric + Anthropic prompt library + LangChain hub, plus the persona-and-domain-priming literature for Qwen-class small models) confirmed two things at once. Other coding agents do explicit persona-and-scope openers; the empirical literature (Zheng et al. arXiv 2311.10054, Wharton "Playing Pretend" report, OpenReview "brilliant mathematician") shows persona is null-to-negative on closed-form tasks for this exact model class. The literature flagged domain-context priming as the candidate addition for empirical validation; Track A below was that test.

Phase 12 entered with the literature-derived hypothesis that domain-context priming would benefit small Qwen-class models, yet Track A (#160, shipped as #168) returned a null result across ninety deterministic reps (3 recipes × 3 variants × 2 models × 5 reps). The domain-priming variant failed to beat the control in any of the six (model, recipe) comparisons and regressed on commit-message tasks because the preamble exceeded the character budget. Persona negative-controls matched or underperformed the control, aligning with Zheng et al. arXiv 2311.10054, while standard deviations remained zero due to deterministic Ollama outputs. The data suggests preamble length is the load-bearing variable rather than the semantic category of the priming. Phase 12 therefore ships the rejection-rationale documentation (#167) and the audience-filter port from sst/opencode (#165) unconditionally. The future-recipe convention in Track B (#161) reduces to the identity-and-scope opener and the flat-YAML inputs piece.

The phase is deliberately lighter than the local-brain-equivalent overhaul the user explicitly ruled out. Sub-tracks:

- ✅ **Track A — Domain-priming experiment (validation gate)** *(shipped #168 — **null result**, domain-priming NOT adopted)*. Built three variants of `commit-message.md`, `prompts/file-summary.md`, and `summarise-issue.md`: (1) current, (2) current plus a one-line domain stipulation at the top of `## Prompt template` ("The input is a unified diff hunk plus three recent commits as shape anchors"), (3) current plus a "You are an expert X" persona prefix as a negative-control. Ran against `qwen3.6:35b-a3b-q8_0` and `qwen3-coder:30b-a3b-q8_0` at `--reps 5` and scored with existing T4 plus new T7 (`file-summary`) and T8 (`summarise-issue`) deterministic scorers. The acceptance gate (variant 2 beats variant 1 by ≥0.5 mean score with non-overlapping stdev intervals; variant 3 does not exceed variant 2) was not met on any of the six (model, recipe) comparisons. Full results under `experiments/results/2026-05-22-phase-12-track-a-domain-priming.md`. The two reusable T7/T8 scorers stay in the repo as infrastructure for future recipe-iteration probes that need a similar structural rubric.
- **Track B — Future-recipe convention (lazy migration).** Update `prompts/README.md` "Adding a new recipe" with an optional convention: a one-sentence identity-and-scope opener at the start of `## Prompt template` (NOT persona — task plus forbidden actions, consolidating directives that today are scattered under `## Anti-hallucination guards`) and an optional `inputs:` block in frontmatter for type hints that `delegate.sh --recipe NAME` can validate pre-flight. The `inputs:` block is constrained to flat `key: type` pairs only (no nesting, no anchors, no flow style) so `awk` can parse it without pulling in `yq` and breaking the "two bash scripts" rule — heavier YAML features stay rejected by design rather than imported by accident. New recipes adopt the convention. Existing recipes migrate only when touched for other reasons. No bulk reformat — the calibration cost of re-running every scorer is real, the user explicitly bounded effort, and the literature does not predict a quality uplift large enough to justify it.
- ✅ **Track C — Document rejection rationale** *(shipped #167)*. Add a section to `prompts/README.md` explaining why this library does NOT adopt the "you are an expert X" persona shape (Zheng et al. null result on Qwen-class), Microsoft Prompty's wholesale schema (calibrated for VS Code Copilot tooling this skill doesn't consume), or fabric's explicit-item-count discipline (small-model fabrication trigger; no measured benefit on the T4/T5/T6 fixture set). Closes the gap before a future contributor drifts toward the public idiom unknowingly.
- ✅ **Track D — Port the strongest external example (sst/opencode audience-filter)** *(shipped #165)*. Sst/opencode's `.opencode/command/changelog.md` slash-command has 17 numbered verbatim-preservation `Rules:` and an audience-filter for "skip commits that are entirely internal, CI, tests, refactors, or otherwise not user-facing." That audience-filter is the specific gap in `prompts/release-note.md`. One-recipe port study: extract the audience-filter rule, integrate into `release-note.md`, dogfood on the next release-notes pass, record verdict.

## Out of scope
