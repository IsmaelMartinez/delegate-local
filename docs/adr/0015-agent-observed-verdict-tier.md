# 15. Agent-observed verdict tier

Date: 2026-06-15

## Status

Accepted.

## Context

Recipe coverage — the share of delegations that carry a HIT/MISS verdict — is the calibration signal the whole skill self-corrects on. Phase E's target is under 20% untracked. The 2026-06-15 measurement found the trend going the wrong way: corrected recipe coverage was 84% lifetime but 78% over the last 14 days and 73% over the last 7. The cause is structural. The boundary hook (#277) and the workflow skills made delegation increasingly automatic — commit messages via the boundary hook, maintainer replies via `/address-pr-comments`, roadmap status via `/roadmap` — while verdict recording stayed manual. The untracked backlog over seven days was 22 successful recipe delegations, every one a skill-driven side effect that recorded no verdict, and the interactive session-end sweep (`scripts/verdict-sweep.sh`, 2026-06-11) that was meant to catch them is not run because it needs a tty and a maintainer who remembers.

The decisive observation is who makes the decision. All 22 untracked delegations were agent-driven: the agent both requested the delegation and decided whether to use the result, often while the maintainer was not watching (many run as background jobs). Auto-delegation moved the decision-maker from the human to the agent. The verdict source has to move with it, or these delegations stay permanently untracked — no amount of nagging the human recovers a verdict for a decision the human never participated in.

But the hit/miss loop has always been a human taste judgment: did the maintainer keep the delegated output, or rewrite or discard it. The agent cannot supply that. What the agent can supply, accurately and without speculation, is a fact about its own behaviour: did it ship the delegated text as-is, or substantially rewrite or discard it. That maps onto hit and miss, but it is a weaker and different signal — "the output was usable enough that the agent shipped it" rather than "the maintainer judged it good." The week before, ADR 0007's #306 fix established the governing principle: coverage and quality are different axes, and coverage must never be bought by faking quality.

## Decision

Agent-observed verdicts are a separate tier, not merged into the human hit-rate. A feedback row records `verdict_source:"agent"` when the agent auto-recorded it; a human verdict omits the field (so it is indistinguishable from the pre-tier legacy rows, which are also treated as human). Coverage and the "untracked" diagnostic count both tiers, so the untracked rate falls as the agent tier fills in. The headline hit-rate is computed by positive selection of human verdicts only, so the quality signal stays exactly as trustworthy as it is today. Every consumer that joins feedback rows to a hit-rate learns the partition: `metrics-summary.sh` and `experiments/quality-trend.py` surface the agent tier as its own coverage-and-usage figure — "how often the agent shipped the delegated output unchanged" — clearly labelled so it is never read as a quality number (the agent column appears only when agent verdicts exist, so single-tier files print unchanged), and the MCP server's `recommend_prompt` (`mcp/delegate_local_mcp/server.py`) likewise counts `hit_count`/`miss_count` over human verdicts only and reports `agent_hit_count`/`agent_miss_count` separately, so the recipe hit-rate it serves to non-Claude clients (Codex, OpenCode, Cursor) cannot be inflated by the agent tier.

The verdict source is a `Stop` hook (`scripts/delegate-verdict-stop-hook.sh`). The agent only knows whether it used an output while it is still running, which rules out a `SessionEnd` hook (the agent is gone — it could only write a blind provisional verdict) and rules out leaning harder on the interactive sweep (no tty on a background job). A `Stop` hook fires when the main agent finishes a turn, the agent is still alive, and the hook can hand it the session's untracked delegations to verdict before it stops. The hook reuses `verdict-sweep.sh`'s base untracked-set join (delegate rows with `exit_status == 0` and no referencing feedback row, inside the window) plus a `.project` filter and minus the tty prompt. It re-engages the agent with `{"decision":"block","reason":…}` — plain `additionalContext` does not reliably re-engage a stopping agent — and instructs it to record each recognised delegation with `delegate-feedback.sh --ts <ts> --source agent hit|miss`, leaving unrecognised ones for the human sweep.

There is no settle floor on the Stop path. By the time a Stop fires the turn is complete and the agent has already used or rewritten the output — that is precisely the moment it can judge honestly from live memory. Excluding sub-floor delegations would push the agent's own work to a future session with no memory of it, recreating the gap one session later.

The loop guard is a session-once marker keyed on the `Stop` payload's `session_id`. Because `decision:"block"` re-engages the agent, an unguarded hook would loop when the agent declines or `delegate-feedback.sh` errors: the next Stop sees the same untracked rows and blocks again to the turn limit. The first time the hook surfaces a batch it writes a marker file (under the metrics directory, the one writable location the scripts already use); every later Stop in the same session sees the marker and exits cleanly without re-injecting. The marker is written only when a batch is actually surfaced, so a session that delegated nothing leaves no marker. There is no `enforce` mode — coercing a verdict is both hostile and dishonest, and a forced verdict is not a fact.

## Consequences

Three honesty properties keep the agent tier out of the quality signal. The recorder defaults to `--source human`, so an accidental omission records as human (the conservative direction for the pre-existing corpus); the Stop hook therefore always passes `--source agent` explicitly, and a test asserts the injected command contains it. The reporting partition is the load-bearing guard: even a mis-tagged agent verdict cannot reach the human hit-rate, because that number is computed by positive selection of `verdict_source == "human"` (legacy no-field rows mapped to human), not by excluding agent. And the agent's instruction frames the judgment as "did you use it," not "was it good," so the recorded fact matches the tier's stated meaning. The tier also reaches telemetry: the OTel feedback span carries `delegate.feedback.source` as unconditional metadata (ADR 0007 changelog), so an OTel-backed dashboard can reproduce the human-only partition rather than re-contaminating the signal one layer downstream.

The untracked diagnostic sharpens rather than blurs: a delegation is untracked only if neither the human nor the agent has judged it, which is the number Phase E cares about driving under 20%. The risk to watch after rollout is partition leakage — the loop returns to measurement to confirm the untracked rate falls as the agent tier fills in while the human hit-rate stays flat; a drop in the human number would mean the partition leaked. The full design rationale, including the two rejected alternatives (a unified hit-rate, and a surface-only mechanism that writes nothing without an explicit answer), lives in `docs/research/2026-06-15-phase-e-verdict-automation-design.md`.

## Update (2026-06-18) — the inline verdict path was already agent-sourced

The Context above assumed "the hit/miss loop has always been a human taste judgment." The live metrics show that was wrong, and the partition leaked from the start — not arithmetically, but at the source. Of the 619 feedback rows tagged human (the default), 616 were recorded inside `delegate-feedback.sh`'s 300-second stale window, at a median of 20 seconds after the delegation they score. That is the inline path SKILL.md instructs the *agent* to run ("Always record the verdict after the call"), not a maintainer reflecting later. Only 3 verdicts came through the long-gap interactive sweep a human realistically uses, and all three are misses. So the headline 81% "human hit-rate" was overwhelmingly the agent's own ship-rate — the very fact this ADR created the agent tier to capture — mislabeled as human taste because the inline command carried no `--source` and the recorder defaulted to human. The reporting partition held (agent-tagged rows never reached the human number), but it was guarding the wrong door: the agent's verdicts were entering as human upstream of the guard.

Two corrections shipped 2026-06-18. The historical data was backfilled: every feedback row with no `verdict_source` whose verdict was recorded within 300 s of its delegation was retagged `verdict_source:"agent"` (616 rows; backup at `metrics.jsonl.bak-2026-06-18`), and the three long-gap rows were left as human. After the backfill the genuine human-quality sample is 3 rows — effectively empty. The inline recording path was then aligned with reality: SKILL.md's verdict instruction and `delegate.sh`'s verdict nudge now pass `--source agent`, so future inline verdicts tag correctly. The recorder default stays `human` so a maintainer recording a real taste judgment (or a legacy row) is unaffected, and the Stop hook becomes the backstop for verdicts the agent did not record inline rather than the only agent-tier source.

This inverts Phase E's binding constraint. Coverage is no longer the problem — with the inline path and the Stop hook both feeding the agent tier, it is near-total. The problem is now the quality signal: the producing agent grading its own output is biased toward "I used it, so it was good," and there is no human sample left to anchor against. Closing that gap is the open question. The unbiased levers already in the repo are the deterministic output checks (ADR 0014's `checks_failed`, an objective structural-quality proxy that cannot hallucinate) and a closed-form independent verifier in the `ground-check` shape (claims → SUPPORTED / CONTRADICTED / NOT-STATED with a deterministic quote backstop) run in a fresh session, with a small periodic human sample used only to calibrate the verifier rather than to grade every delegation. An open-ended "rate this output" local-model judge is explicitly not the answer — it would trade the producing agent's self-bias for a weak-reasoner's noise, the failure mode SKILL.md is built around.
