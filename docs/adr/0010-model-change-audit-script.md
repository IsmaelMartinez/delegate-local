# 10. Model-change audit script — three gates and an ADOPT / INVESTIGATE / REJECT verdict

Date: 2026-05-24

## Status

Accepted.

## Context

The skill routes work through `scripts/pick-model.sh`, whose per-tier preference lists are the load-bearing routing decision. When a new Ollama model lands on the reference host — pulled because `llmfit recommend` surfaced it, or because a new release was announced, or because a user's `scripts/init.sh` run promoted it — the question "should this model be added to a tier's preference list" historically depended on three signals that lived in three different places. The trigger eval (`scripts/eval-skill-triggers.sh --ollama <model>`) answered "does this model still understand the skill's frontmatter description well enough to invoke the right tier"; the empirical scorers (`experiments/score-t4.sh` through `experiments/score-t6.sh`, plus T7 for `file-summary` and T8 for `summarise-issue` when those scorers are present) answered "does this model produce structurally-correct output on the recipe shapes the skill actually serves"; and chat-template inspection (`ollama show --modelfile <model> | grep -A20 '^TEMPLATE'`) answered "does this model's chat template carry surfaces the dispatch wrapper does not handle".

The user's auto-memory at `project_model_family_transition_risks.md` captures the load-bearing risk: chat-template silent degradation plus 0.4-0.6 F1 drops on structured extraction have been measured during model-family transitions where the candidate passed `llmfit recommend`'s hardware-fit predictor. A model that scores well on llmfit can still ship a chat template with a default system prompt that overrides the recipe's directives, or with think-block wrappers the dispatch wrapper does not strip, or with tool-call bindings that change the response shape — and the failure is silent at the wrapper level because the model still returns text, just text the recipe's scorers reject. The three gates exist precisely because llmfit's prediction does not cover any of those failure surfaces.

Running the three gates manually for every candidate model was the historical practice; the friction was that the manual run did not produce a side-by-side comparison against the current tier incumbent, and the chat-template inspection was easy to skip when the trigger eval and scorers both passed. The Phase 14 Track A brief was to bundle the three gates into one script with one verdict and one set of exit codes so the audit could not silently skip a gate.

## Decision

`scripts/model-change-audit.sh <new-model> [<tier>]` runs all three gates in sequence and prints a single verdict at the end. The script never edits `pick-model.sh` — the user decides whether to promote based on the printed verdict. The model must already be installed (`ollama list` must show the exact tag); the script never pulls. Tier is optional and is inferred from `pick-model.sh --print-prefs` if omitted, with substring matching against the prefs arrays — first tier whose prefs contain a case-insensitive substring match against the candidate wins, falling through to "no incumbent comparison" if no tier claims the model.

The three gates and their pass criteria:

The trigger eval gate runs `bash scripts/eval-skill-triggers.sh --ollama <model>` against `evals/eval-set.json` and captures recall and negative-precision. Pass condition: both metrics ≥ 0.9, matching the threshold the eval-set itself encodes. The gate measures whether the model understands the skill's frontmatter description well enough to invoke the right tier on positive paraphrases and stay out on negative paraphrases.

The empirical scorer gate runs `bash experiments/run-baseline.sh --reps 3 <model>` and then `experiments/score-t4.sh` through `experiments/score-t6.sh` (plus T7 and T8 if those scorers are present in the tree). Pass condition per scorer: mean ≥ incumbent mean if a recent incumbent baseline exists for comparison, else mean ≥ 0.8 as the absolute floor. The gate measures whether the model produces structurally-correct output on the recipe shapes the skill actually serves — the T4 commit-message structural checks, the T5 JSON-shape schema conformance, the T6 regex acceptance tests.

The chat-template gate runs `ollama show --modelfile <model>` and inspects the `TEMPLATE` block. Three template surfaces are detected and annotated but treated as compatible because the dispatch wrapper already handles them: `<think>` blocks (the wrapper sends `think:false`), `chat_template_kwargs.enable_thinking` plumbing (the MLX path binds correctly), and templates that default a system prompt (the wrapper sends user-role only, so there is no implicit override). The fourth surface — tool-call bindings (`<|tool_call|>`, `<tool_call>`, `tool_calls`) — is the only one that flips `template_compatible=0` and so propagates to the verdict, because the wrapper does not bind `tool_calls` and the response shape would change. The gate is the loudest of the three for that fourth surface specifically, because chat-template divergence on tool-call bindings is the silent-failure shape — the model still returns text, the wrapper still considers the call successful, but the recipe's calibration no longer binds.

The verdict and exit-code mapping is the API the audit-script exposes for downstream tooling:

`ADOPT` (exit 0) — all three gates pass. The new model is safe to add to its tier's preference list at the position the user chooses. The audit does not promote; the user runs `git diff scripts/pick-model.sh` themselves.

`INVESTIGATE` (exit 1) — marginal degradation. Any gate drops less than 0.1 below the incumbent but stays above the absolute floor. The verdict means "not safe to adopt without understanding why one gate weakened, but not categorically broken". A common shape: T5 JSON drops from 1.00 to 0.93 because the model's chat template adds a leading explanatory sentence the scorer's JSON-fence-strip cannot remove; the model is still useful but the JSON recipe needs a re-anchoring pass before adoption.

`REJECT` (exit 2) — material degradation. Any scorer mean drops more than 0.1 below the incumbent, any trigger metric drops below the 0.8 absolute floor, or the chat template diverges on a surface the wrapper does not handle (today: tool-call bindings only — `<think>` blocks, `enable_thinking` plumbing, and default system prompts are all annotated but treated as compatible because the wrapper neutralises them). The verdict means "do not add to the preference list". The reject-on-template-divergence rule is the central calibration-protection move — a model that produces correct output via a divergent template will produce broken output the moment a recipe relies on a directive the divergent template overrides.

`error` (exit 3) — gate execution failed. Model not installed, scorers crashed, baseline-script unavailable, etc. Distinct from REJECT so downstream tooling can branch on "the audit could not run" versus "the audit ran and the model failed".

The four-way exit code mapping (0/1/2/3) matches the pattern used in `scripts/apply-and-test.sh` (which encodes six outcomes 0-5 the same way) so callers consume the verdict by checking the exit code without parsing stdout — the same convention this skill uses across its other machine-parseable surfaces.

The script grew from the Phase 14 Track A discussion in issue [#198](https://github.com/IsmaelMartinez/delegate-to-ollama/issues/198) and shipped via PR [#200](https://github.com/IsmaelMartinez/delegate-to-ollama/pull/200) (squash `72e883d`, merged 2026-05-23) with 41 unit tests in `tests/test-model-change-audit.sh` covering usage errors, model-not-installed, tier inference, the three verdict paths, incumbent-baseline comparison, and the no-inference fallback. The same PR refactored `scripts/pick-model.sh` to add a `--print-prefs` flag so the audit script and `scripts/init.sh` consume tier definitions from one source rather than re-parsing the case statement separately.

The rejected alternative was extending `scripts/audit-models.sh` (the existing llmfit-driven advisory script) to also run the three gates. Rejected because `audit-models.sh` is a read-only suggestion surface that surveys the installed model set and prints upgrade suggestions; it runs against every installed model and is cheap to call. Embedding the three gates into it would have inverted the cost profile — the trigger eval alone takes 10-30 seconds per model, the empirical scorers another minute or two, and running the chat-template gate against every installed model would have made the audit script slow enough that users would stop running it monthly. Splitting the per-candidate validation into a separate script kept `audit-models.sh` cheap and made `model-change-audit.sh` the deliberate per-candidate gate.

## Consequences

Model upgrades that pass `llmfit recommend` can no longer silently degrade the recipe library. The three-gate audit catches the failure shapes llmfit's hardware-fit predictor does not cover: trigger drift, structural-output regression, and chat-template incompatibility. The user retains the promotion decision — the script is read-only with respect to `pick-model.sh` — but the decision is now informed by side-by-side numbers against the current incumbent rather than against an unstated mental baseline.

The chat-template inspection is the central calibration-protection move. The other two gates can be re-run easily and their failures are visible (low recall, low T5 mean, etc.); chat-template divergence is the silent-failure surface that the calibration loop cannot catch automatically because the model still returns text and the metrics row still shows status=0. Promoting a chat-template-divergent model to a tier preference would invalidate every recipe that relies on a directive the divergent template overrides, and the invalidation would only surface as a stream of fresh MISS reports that the recipe authors would then have to chase. The `REJECT` verdict on template divergence stops the invalidation at the audit step rather than after the broken model has been routed to.

The script's read-only-with-respect-to-the-repo invariant is load-bearing for the audit's trust model. A script that auto-edits `pick-model.sh` based on a verdict would invite "I ran the audit and now my routing is different" surprises; the explicit "the user decides whether to promote based on the printed verdict" rule keeps the promotion path manual and visible.

What would justify revisiting the design: a chat-template surface the current grep-based inspection misses (a new template tag that overrides recipe directives in a way that doesn't trip the existing checks), a recipe shape whose structural scorer doesn't yet exist (the Phase 14 Track C deferral covered `pr-review-reply` and `polish-reply` for exactly this reason — their outputs are free-form prose whose correctness depends on context the scorer cannot independently verify), or a model class whose verdict the four-way ADOPT/INVESTIGATE/REJECT/error mapping cannot express cleanly. The first justifies extending the chat-template gate; the second justifies adding a new T-scorer when a structural anchor becomes visible; the third justifies extending the verdict vocabulary, which would be a breaking change to the exit-code API and so should be deferred until a real model class surfaces the need.

The cost of this ADR is one more file in `docs/adr/`. The benefit is that the rationale for splitting per-candidate validation off from `audit-models.sh`, the chat-template-as-silent-failure-surface argument, and the exit-code-as-machine-API pattern now live in a durable single artifact, so a future contributor who wonders why the audit script exists separately from the monthly llmfit advisory finds the trade-off documented here.
