# 22. Code-generation fan-out with a test oracle

Date: 2026-06-19

## Status

Accepted. Builds directly on ADR 0018 (the fan-out negative result) by changing the one variable that made 0018 negative — the absence of a trustworthy selector — and on ADR 0020 (the verify-and-escalate gate), which it reuses as the failure fallback. Scope is Consumer A of the delegation-rethink initiative: single-file code patches selected by a director-provided test. The debiasing second-opinion layer (Consumer B) is a separate, later ADR.

## Context

ADR 0018 shelved sampling fan-out because drawing N samples bought no quality: the local MLX server ignored per-request seeds, so the samples came back byte-identical, and even with diversity restored the prose fixtures (T5, T6) sat at their single-shot ceiling with no selector to convert variety into a better answer. Two things changed. The seed bug is fixed and validated upstream (mlx-lm PR #1331, tracked as issue #323), so fan-out now produces genuinely diverse samples on Ollama today and on MLX once the fix ships. And the deeper lesson from 0018 — fan-out only pays when a trustworthy selector picks the winning sample — points at the one task class that has a perfect, automatic selector: code patches, judged by a real test.

## Decision

A new orchestrator, `scripts/fanout-patch.sh`, owns the fan-out decision logic and nothing else. It composes two trustworthy pieces that stay untouched and independently tested: `scripts/apply-and-test.sh` (the oracle — apply a patch to a throwaway copy, run pytest, emit a verdict) and `scripts/delegate.sh --recipe fix-with-test` (generation), made diverse by a new `DELEGATE_SEED` passthrough at temperature above zero. `pick-model.sh` resolves the tier transitively, through delegate.sh on each call, which keeps the one code model warm across the N back-to-back generations.

The flow is a single pass with a fallback. The director hands the orchestrator a source directory and the real failing test. For seeds one through N the orchestrator draws a candidate patch from the warm code model and runs each through the oracle. If at least one candidate makes the test pass, it returns the smallest passing diff, a tie-break toward the least invasive fix. If all N fail, it escalates a small fixed number of samples to a strong tier through the same oracle; if one passes, it returns that. If nothing passes, it hands back the closest failing attempt with a clear "no passing patch" verdict. The invariant across every path is that nothing is returned as done unless it actually passed the test in a clean copy, so the worst case is "no patch", never "a broken patch that looks fine".

Five decisions from the design conversation are load-bearing. The oracle is a director-provided real failing test, not a generated one, because a wrong generated test silently certifies a wrong patch. The patch surface is a single file for the first version. Diversity comes from N seeds on one warm code model rather than multiple models. The failure path escalates to a strong model and then hands back, never shipping an unverified patch, which is exactly the ADR 0020 escalate-then-handback shape reused here. And when a majority of samples refuse (the recipe gives each one a `REFUSE:` hatch for a self-contradictory test), the orchestrator hands back "the test may be wrong" rather than burning an escalation trying to satisfy a broken oracle. The oracle is authoritative over the prose: a patch wins if and only if it passes the test, regardless of what its prose said, and a passer that coexists with a refuse-majority is flagged so the director sanity-checks the test. Because the seed only works on Ollama until #1331 ships, the fan-out defaults to the Ollama backend.

## Measurement

The go/no-go is the measured lift: best-of-N pass-rate minus single-shot pass-rate over a fixture suite with genuine single-shot headroom. A lift of roughly 0.2 to 0.3 earns the right to widen; below that, on a suite that genuinely has headroom, the mechanism would be shelved with the evidence written down the way cheap-first was (#322). The load-bearing caveat is that the fixtures must have single-shot headroom: if the code model already nails every fixture single-shot, best-of-N cannot lift anything and concluding "no value" would be false, exactly the T5/T6 trap.

That caveat is what the first reading hit. Five easy single-function fixtures (off-by-one, wrong operator, missing edge, fencepost, accumulator) at temperature 0.7 over three reps returned single-shot 1.000, best-of-5 1.000, lift 0.000, 7.5 s per fix, with no escalation and no handback. Every patch was oracle-verified, so the mechanism is correct, but there is no headroom because `qwen3-coder-next:latest` passes them all single-shot. Sharpening to four deliberately harder fixtures (numeric version compare, Roman numerals with subtractive notation, recursive flatten, interval-merge-without-sort) at temperature 1.0 over five reps returned the same shape: single-shot 1.000, best-of-5 1.000, lift 0.000, 12.5 s per fix. Both runs are under `experiments/results/2026-06-19-fanout-patch*.md`.

The go/no-go is therefore deferred, not decided. The mechanism, the oracle, the escalation path, and the refuse and no-pass handbacks all function correctly (the 19 unit assertions plus the live runs where every emitted patch passed its test), and the cost is bounded and cheap. What is unproven is whether fan-out pays, because the natural primary sits above the 40-to-80-percent band on single-function fixes where best-of-N could convert diversity into recovered passes. Demonstrating the lift needs a measurement with real headroom, and the lineup's capability cliff (#322: the 0.6B is below the floor on every fixture, the 30B above it, nothing competent between) blocks manufacturing it with a weaker primary on this task class. The two named paths are a mid-tier 4-to-8B primary (the same gap #322's cheap-first work waits on) or a harder, multi-step task class than single-function bug fixes. Until one produces a measured lift, the widenings stay gated and the mechanism ships as correct-and-cheap-but-lift-unproven.

## Consequences and honest boundaries

The hard boundary is safety. The oracle runs pytest with full user privileges and no sandbox, which is fine for the one use it is designed for and dangerous for anything else, so `fanout-patch.sh` is strictly for the director's own author-controlled source and tests over locally-chosen models — never untrusted source, an externally-supplied test, or model output from a non-local source. That constraint is refused at the boundary, not left implicit, the same rule `apply-and-test.sh` already states.

Cost is bounded by construction: N cheap samples plus a small fixed escalation count, each capped by the oracle's own pytest timeout. The mechanism widens what we delegate from text to code, but only on the narrow task class where a hard oracle exists; the moment the test is one the director cannot trust or re-run, the discipline that makes this safe is gone and the work belongs back with the director. The natural widenings once it pays — multi-file patches, then a few seeds across a couple of models for architectural diversity — are deliberately deferred, each gated on this one paying first.
