# 8. Recipe `inputs:` frontmatter — flat YAML, no nesting, no yq dependency

Date: 2026-05-24

## Status

Accepted.

## Context

Phase 12 Track B (issue [#161](https://github.com/IsmaelMartinez/delegate-local/issues/161)) operationalised the future-recipe convention that the 2026-05-22 prompt-library research sweep arrived at. The sweep surveyed sst/opencode, QwenLM/qwen-code, Microsoft Prompty, fabric, the Anthropic prompt library, and LangChain hub, then layered the persona-and-domain-priming literature on top. Two of Track B's three pieces were uncontroversial (the one-sentence identity-and-scope opener at the top of `## Prompt template`, and the rejection-rationale documentation that became Track C). The third piece was a type-validation surface for `--var key=value` inputs, so a caller passing `--var pr_number=foo` against a recipe that expects an integer fails fast with a named-key error rather than silently producing a poorly-formed prompt that wastes both wall time and electricity on a deterministic miss.

The candidate that informed the design was Microsoft Prompty, which carries a richer YAML frontmatter shape with nested input definitions, default values, and Jinja-style templating. Adopting Prompty wholesale would have given the validation surface for free, but at the cost of a YAML parser. `awk` and bash 3.2's pattern matching cannot parse nested YAML safely (block scalars, anchors, flow style, multi-line values), so the implementation would have needed `yq` (or a Python helper) on every host, breaking the project's "two bash scripts" invariant. The invariant is load-bearing because the skill ships as a Claude Code plugin / `npx skills add` install where the runtime is bash plus `jq` plus `awk` plus `perl` only — adding a YAML runtime dependency would have closed off macOS-shipped-bash hosts and Linux containers that lack `yq` in their base image.

`prompts/README.md` Convention 2 captures the surface the validator implements; the validator itself lives in `scripts/delegate.sh` around lines 487-635 (the awk extractor at 496-503, the per-input type-check loop at 511-541, the `--var` validation at 547-585, the `{{stdin}}` integration at 593-614, and the missing-required check at 620-634).

## Decision

`scripts/delegate.sh --recipe NAME` parses an optional `inputs:` block from the recipe's YAML frontmatter and validates each provided `--var key=value` against the declared type before contacting the model. The block is constrained to flat `key: type` pairs only — no nesting, no anchors, no flow style, no block scalars, no multi-line values. The supported type vocabulary is four tokens exactly: `integer`, `string`, `integer?`, `string?`. The `?` suffix marks the input as optional (absent `--var` is allowed; if provided, the value is still type-checked).

The awk extractor at `scripts/delegate.sh:496-503` is the precise gate: it recognises only lines matching `^[[:space:]]+[a-zA-Z_][a-zA-Z0-9_]*:[[:space:]]*[a-zA-Z?]+[[:space:]]*$` inside an `inputs:` block bounded by the frontmatter `---` markers. Lines that contain nested structure, flow syntax (`{foo: bar}`), arrays, or block-scalar indicators (`|`, `>`) silently fail to match and are dropped — they do not crash the parser, they simply are not extracted as type declarations. The type-check loop at lines 511-541 builds parallel indexed arrays (`declared_keys`, `declared_types`, `declared_optional`) because bash 3.2 has no associative arrays, so the matcher uses an O(n*m) linear scan that is fine for the handful of inputs a recipe declares in practice. Validation runs BEFORE placeholder substitution so the caller sees the type error rather than an opaque "missing placeholder" downstream message; the validator exits 2 with a stderr line naming the offending key, the expected type, and the actual value (e.g. `delegate: --var pr_number expected type 'integer', got 'foo'`). The unsupported-type-declaration path at line 533 additionally names the recipe, because that error is the recipe author's bug rather than the caller's.

The `{{stdin}}` placeholder maps to a declared `stdin: string` input. A recipe can require piped context via the typed surface without forcing the caller to pass it twice (once as `--var`, once piped); if the recipe declares a non-string type for stdin, the piped value is type-checked against that declaration the same way `--var` values are. Undeclared `--var` keys pass through untouched — strict-mode rejection of unknown keys is deferred until lazy migration is more complete, so a recipe that declares only `pr_number` does not break callers passing `--var notes=...`. Recipes without an `inputs:` block skip the validation entirely (the back-compat path), so the convention is genuinely opt-in.

The rejected alternative was importing `yq` (or a Python YAML parser) to support the richer Prompty schema. That would have given nested types, default values, enum constraints, and the rest of the YAML-typed-config vocabulary for free. The drawback is that every host running this skill would need `yq` on PATH (a Go binary, not packaged on macOS-shipped systems by default) or a Python interpreter at a known version. The skill's distribution model (Claude Code plugin / `npx skills add` / manual `cp -r`) deliberately assumes the bash-plus-coreutils-plus-jq-plus-perl baseline that ships on macOS and Linux. Adding `yq` to that baseline would have closed off the simplest install paths and added a per-host install gotcha that the existing four-tool baseline avoids.

Track B Phase 12 shipped via PR [#178](https://github.com/IsmaelMartinez/delegate-local/pull/178) (squash `c9e6f8f`, merged 2026-05-22) with the convention documentation, the validator, and one worked example. PR [#194](https://github.com/IsmaelMartinez/delegate-local/pull/194) (squash `542da68`, merged 2026-05-23) backfilled the `inputs:` block onto the 13 existing recipes during the Phase 13 round.

## Consequences

Recipe authors who want pre-flight type validation get it via four type tokens and a flat-pairs grammar. The validator catches the common authoring mistake (passing a non-integer where the template expects one) before any model is contacted, which on the prose tier of a 35B-class model saves 4-10 seconds of wall time per failed call and the corresponding electricity. The named-key stderr surface ("recipe X missing required inputs: pr_number body") lets the caller fix the call in one pass rather than discovering missing inputs one at a time.

The decision pins this skill's validation vocabulary to a deliberately small subset of YAML. Anything richer (nested types, enums, defaults, regex patterns) is rejected by design rather than imported by accident, mirroring the Microsoft Prompty rejection rationale in `prompts/README.md` "What this library does NOT adopt". Recipe authors who want a richer validation surface have to argue for the additional grammar against the "two bash scripts" rule, which makes drift toward Prompty's full surface a deliberate decision rather than a creeping default.

The lazy-migration path matters operationally. Existing recipes did not need a bulk reformat to gain validation — they got it incrementally as each was touched during Phase 13's anti-padding canonicalisation pass (PR #195) and the per-recipe `inputs:` backfill (PR #194). New recipes adopt the convention; old recipes keep working unchanged. The undeclared-`--var`-pass-through rule is the load-bearing detail that makes partial migration safe — a recipe that declares one input does not break callers passing extra inputs the declaration does not mention.

What would justify revisiting the decision: a recipe authoring use case that the flat-pairs grammar genuinely cannot express, where the workaround (multiple recipes, callsite logic to pick between them, or string-encoded composite values) is uglier than the per-host `yq` install would be. The current 13 recipes did not surface such a case during the Phase 13 backfill; the four type tokens covered every observed `--var` shape. If a future recipe needs (say) a list of allowed values, the first move is to encode the constraint in the recipe's `## Variables` section as documentation and let the type-check stay at `string`; if that proves insufficient across multiple recipes, the second move is to add a fifth type token (`enum: [a, b, c]`) to the awk extractor; the third move is the `yq` import, and only after both lighter moves have been measured to fall short.

The cost of this ADR is one more file in `docs/adr/`. The benefit is that the empirical rationale for the flat-pairs grammar and the explicit rejection of nested YAML now live in a durable single artifact, so a future contributor who wonders why the validator does not handle their nested case finds the trade-off documented here rather than having to re-derive it.
