# 12. `flaky_on_models:` frontmatter — structural tier-gate before the pre-flight canary

Date: 2026-05-24

## Status

Accepted.

## Context

`prompts/pr-description.md` accumulated calibration evidence between 2026-05-10 and 2026-05-13 that the recipe stalls or produces massive hallucinations on 35B-class prose-tier models regardless of body size — measured on Ollama and on MLX, on inputs ranging from 612 bytes to 5.2 KB. Issue [#110](https://github.com/IsmaelMartinez/delegate-local/issues/110) documented stalls of 6-10 minutes on the resolved prose-tier model; subsequent comments narrowed the discriminator to model parameter count at recipe-sized prompts. The recipe's calibration notes named hand-writing as the active mitigation since 2026-05-10. The Phase 11 round-3 retrospective surfaced `pr-description` as the standout weak recipe in the 22-day trend report at a 45% hit rate (5 HIT / 6 MISS across 11 verdicts); the documented mitigation existed but the wiring to enforce it did not.

The Phase 11 round-3 entry deferred the wiring under "not urgent". Phase 16 Track A shipped the wiring as a generic recipe-authoring convention. The convention is documented in `prompts/README.md` Convention 4 with two recipe-authoring rules: anchor every listed substring in measured evidence (a `flaky_on_models:` entry is a structural claim that this recipe will not work reliably on this model class, and the recipe's calibration notes must cite the empirical observation that justified the entry), and list substrings (not exact model names) so the gate adapts across quantisation suffixes and naming-convention drift.

The implementation lives in `scripts/delegate.sh` lines 731-775. The awk extractor at lines 732-744 parses the YAML frontmatter `flaky_on_models:` block as a list of dash-prefixed entries; the case-insensitive substring match at lines 746-755 folds both the model name and each pattern to lower case and tests for substring containment with bash's `[[ "$model_lower" == *"$pat_lower"* ]]`; the refusal at lines 756-773 writes a metrics row tagged `exit_status:4`, emits an OTel span carrying the same status, prints a stderr message naming the recipe, the resolved model, the matched pattern, and three recovery options, then exits 4.

A separate concern shaped the gate's position relative to the pre-flight canary added in issue [#110](https://github.com/IsmaelMartinez/delegate-local/issues/110)'s mitigation work. The canary (a 1-token probe with `curl --max-time ${DELEGATE_PREFLIGHT_TIMEOUT:-10}` against the resolved model) catches the dynamic failure mode — model unreachable, model cold-loading, model stalling at load time. The flaky-gate catches a different failure mode entirely: the model responds promptly and produces output, but the recipe's calibration history has documented that the output is empirically unreliable on this model class. The two gates address orthogonal failures, and their ordering matters because they have different signal-to-noise trade-offs.

## Decision

`scripts/delegate.sh --recipe NAME` parses an optional `flaky_on_models:` list from the recipe's YAML frontmatter and refuses with exit 4 before dispatching the request when the resolved model name contains any listed substring (case-insensitive). The gate runs BEFORE the pre-flight canary because the refusal is structural ("this recipe will not work reliably on this model class") rather than dynamic ("the model isn't responding right now") — no point probing a model the recipe already classifies as unreliable. Skipping the canary on a flaky-matched model also avoids paying the canary's wall-time cost (up to `DELEGATE_PREFLIGHT_TIMEOUT` seconds, default 10) on a request the wrapper has already decided to refuse.

The matching algorithm is case-insensitive substring containment, not exact model-name equality. Model names vary by quantisation suffix (`-q8_0`, `-q4_K_M`), backend (Ollama uses `:`, HuggingFace uses `/`), and underscore/hyphen conventions across releases. A substring like `qwen3.6:35b` matches `qwen3.6:35b-a3b-q8_0`, `qwen3.6:35b-instruct`, and (case-insensitively) `Qwen3.6:35B`. Recipe authors list all naming conventions observed in real `pick-model.sh` resolution so the gate adapts across upgrades without needing a recipe edit for every new quantisation tag. Full quantisation suffixes are explicitly NOT listed — they change too often for the gate's substring claim to stay accurate.

The opt-out env var `DELEGATE_FORCE_FLAKY=1` sends the request anyway. The override exists specifically so callers can capture fresh evidence that the flaky-class behaviour has changed across model upgrades. If a future Qwen3.7 release behaves better on the pr-description shape, the override lets the caller measure the new behaviour, record the result via `delegate-feedback.sh hit|miss`, and then update the recipe's frontmatter — rather than silently ripping the gate out and losing the calibration history. The override is per-call rather than per-shell or persistent; the failure mode of "I forgot it on" is bounded to the operator's current invocation.

The refusal-emits-metrics row design is the central observability hook. Writing the JSONL line with `exit_status:4` (and the would-be `prompt_chars` from the templated request) means `scripts/audit-metrics.sh` can pivot on flaky-gate refusals later — the same way it pivots on `exit_status:3` rows from canary timeouts. Without the metrics row, the refusal would be invisible to the calibration loop and the gate's effectiveness would be unmeasurable. The OTel span emission mirrors the same status so observability backends with the OTLP exporter enabled see the refusal alongside the successful dispatches.

The rejected alternative was wiring the gate AFTER the pre-flight canary. Rejected because the canary's purpose is to verify the model is reachable and responding — for a recipe-and-model pair the frontmatter already classifies as flaky, the canary's signal is irrelevant. If the canary succeeds, the wrapper still refuses; if the canary fails (timeout), the wrapper would refuse with the canary's exit-3 instead of the flaky-gate's exit-4, losing the structural-vs-dynamic distinction in the metrics row and confusing downstream audit logic. Sitting the flaky-gate first preserves the clean exit-code mapping (3 = dynamic, 4 = structural) and avoids paying the canary cost on a doomed-by-policy request.

The second rejected alternative was exact-model-name matching instead of substring containment. Rejected because the quantisation-suffix problem makes exact matching brittle. A recipe that lists `qwen3.6:35b-a3b-q8_0` exactly would silently fail to gate `qwen3.6:35b-a3b-q4_K_M` (same model, different quant), `qwen3.6:35b-instruct` (same parameter count, different tag), and `Qwen3.6:35B-A3B-Q8_0` (MLX cache convention with HuggingFace casing). The substring approach binds on the parameter-count claim (which is what the recipe is actually classifying) and adapts across the naming-convention drift around it.

`pr-description.md` is the worked example shipped with Phase 16 Track A. Its `flaky_on_models:` block lists `qwen3.6:35b` and `qwen3-next:80b` plus the underscore-and-dash naming variants — each anchored in the 2026-05-10/11/12/13 stall measurements the recipe's calibration notes already document. Empirical verification on the reference host: prose tier (qwen3.6:35b-a3b-q8_0) refused with exit 4 plus the named-recipe-plus-recovery-options stderr; long-context tier (qwen3-next:80b-a3b-instruct-q8_0) refused with exit 4; code tier (qwen3-coder-next:latest) passed the gate and dispatched normally. 21 new test assertions in `tests/test-delegate.sh` cover the match / non-match / back-compat / override / case-insensitive paths.

Phase 16 Track A shipped via PR [#209](https://github.com/IsmaelMartinez/delegate-local/pull/209) (squash `d03c14f`, merged 2026-05-24).

## Consequences

Recipes with documented empirical evidence of unreliable behaviour on a specific model class have a structural enforcement mechanism rather than only an advisory calibration note. The 45% hit rate on `pr-description` that the Phase 11 round-3 trend report flagged stops being a continuous source of bad output the caller has to recognise and discard — the wrapper refuses before any tokens are spent. Callers who hit the refusal get three concrete recovery options in the stderr (hand-write the output, route to a different tier with `--tier code`, override with `DELEGATE_FORCE_FLAKY=1`) so the refusal points at the active mitigation rather than dead-ending the workflow.

The structural-before-dynamic ordering protects the calibration loop from a class of false signals. Without the flaky-gate sitting first, a recipe-and-model pair the frontmatter classifies as flaky would still trigger the canary; the canary would usually succeed (the model is reachable, the load is warm, the probe returns within 10 seconds), and the wrapper would dispatch the doomed request anyway. The metrics row would show a successful dispatch followed by a MISS verdict from the caller, which the calibration loop would interpret as a recipe-iteration signal even though the structural claim is already documented. Refusing before the canary makes the structural claim authoritative: a flaky-matched call NEVER reaches the model, so the calibration history's MISS rate stays anchored to the cases the recipe author has not yet classified.

The substring-with-case-folding matching rule is the operational detail that keeps the gate working across model upgrades. Recipe authors who list `qwen3.6:35b` rather than the full quantisation-suffixed name get gate coverage for every quant variant Ollama or HuggingFace ships, including future variants nobody has measured yet. The trade-off is that an overly broad substring can over-match — listing `qwen` would gate every Qwen model — so the convention rule "list substrings (not exact model names)" is paired with the implicit rule "list substrings narrow enough to claim the parameter-count or model-family axis the recipe is actually classifying". The two-rule convention covers both failure modes.

What would justify revisiting the design: a recipe whose flaky-class boundary is genuinely sub-parameter-count (some 35B-class models work on the recipe shape and others do not) where substring matching produces too many false refusals; a model-naming convention shift that breaks the substring-folding assumption (e.g., a backend that uses semver-suffixed names where `qwen3.6:35b` would match `qwen3.6:350b` accidentally); or a calibration history showing that gated recipes silently re-stale because the override-and-measure-fresh path is not exercised often enough. The first justifies adding a second axis to the frontmatter (`flaky_on_models_unless:` for the per-tag carve-out); the second justifies a more careful matching algorithm (anchored substring, full-word boundary); the third justifies a per-recipe cadence prompt to re-measure gated entries.

The cost of this ADR is one more file in `docs/adr/`. The benefit is that the structural-vs-dynamic ordering rationale, the substring-with-case-folding choice, and the exit-4-with-metrics-row observability hook now live in a durable single artifact. A future contributor who reads the `flaky_on_models:` parser code and wonders why the gate sits before the canary finds the reasoning here rather than only in the inline comment at `scripts/delegate.sh:726-730`.
