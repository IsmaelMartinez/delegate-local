# Phase E — verdict automation design

Date: 2026-06-15. Design document, not an implementation plan and not code. It specifies the mechanism that closes the auto-delegation/verdict gap the 2026-06-15 Phase E measurement surfaced, and records the one architecture-defining decision (separate verdict tier) that was made before drafting. The implementation plan follows in a separate step once this design is approved.

## Problem

The Phase E measurement on 2026-06-15 took the coverage reading and found the trend going the wrong way. Corrected recipe coverage is 84% lifetime — the sub-20%-untracked target is nominally met — but it is 78% over the last 14 days and 73% over the last 7, slipping rather than falling. The cause is structural and ties directly to the boundary-hook and workflow-skill work that shipped earlier: delegation has become increasingly automatic while verdict recording has stayed manual, so coverage drops as auto-delegation rises. The genuine untracked backlog over the last seven days is 22 successful recipe delegations — 11 commit-message, 7 maintainer-reply, 4 roadmap-status — every one a skill-driven side effect that recorded no verdict. The session-end sweep that was meant to catch them (`scripts/verdict-sweep.sh`, shipped 2026-06-11) is not being run, because it is interactive and relies on the maintainer remembering to invoke it.

The decisive observation is who drafts these artifacts. All 22 untracked delegations are agent-driven: the commit message comes from the boundary hook firing on `git commit`, the maintainer reply from `/address-pr-comments`, the roadmap status from `/roadmap`. In each case the agent both requested the delegation and decided whether to use the result, often while the maintainer was not watching (many of these run as background jobs). Auto-delegation moved the decision-maker from the human to the agent. The verdict source has to move with it, or these delegations stay permanently untracked — no amount of nagging the human will recover a verdict for a decision the human never participated in.

## What the agent can honestly report

The hit/miss loop has always been a human taste judgment: did the maintainer keep the delegated output, or rewrite or discard it. The agent cannot supply that taste judgment. What the agent can supply, accurately and without speculation, is a fact about its own behaviour: did it use the delegated text as-is (or with trivial edits), or did it substantially rewrite or discard it. That maps cleanly onto hit and miss, but it is a weaker and different signal — "the output was usable enough that the agent shipped it" rather than "the maintainer judged it good." Treating those two signals as identical would quietly contaminate the one number the whole calibration loop exists to protect. The #306 fix established the governing principle the week before: coverage and quality are different axes, and coverage must never be bought by faking quality.

## Decision: separate verdict tier

The architecture-defining choice — confirmed with the maintainer before drafting — is to keep agent-observed verdicts as a separate tier rather than merging them into the human hit-rate. Concretely, an auto-recorded feedback row carries a `verdict_source:"agent"` field; a manually recorded one carries `verdict_source:"human"` (or omits the field, for rows written before this design — those are treated as human, preserving the existing signal). Coverage counts both tiers, so the untracked rate falls toward the target. The headline hit-rate is computed over human verdicts only, so the quality signal stays exactly as trustworthy as it is today. Reporting surfaces the agent tier separately, as a coverage-and-usage figure, never folded into the taste number.

The two rejected alternatives were a unified hit-rate (simplest, one number, but blends "agent used it" with "maintainer judged it good" — rejected on the same honesty grounds as #306) and a surface-only mechanism that writes no verdict without an explicit answer (most conservative, but it cannot close the gap for background jobs where no one answers the prompt — it reduces to the sweep that is already not being run).

## Mechanism: a Stop hook that hands the batch back to the live agent

The verdict source must be the agent, and the agent only knows whether it used an output while it is still running. That rules out a `SessionEnd` hook (the agent is gone by then — it could only auto-write a blind provisional verdict, which is the rejected unified-blind path) and rules out leaning harder on the interactive sweep (it reads `h/m/s/q` from a tty that a background job does not have). The fit is a `Stop` hook: it fires when the main agent finishes a turn, the agent is still alive, and the hook can inject `additionalContext` instructing the agent to record verdicts before it stops.

The hook is a new thin script, `scripts/delegate-verdict-stop-hook.sh`, modelled on the existing boundary hook's structure and failure posture. On every Stop event it reads the harness payload and fails open on any error or missing `jq` (a verdict sweep must never wedge a session). It derives the current project with the same rule the boundary hook and `delegate.sh` use. It then scans the metrics JSONL for `source:"delegate"` rows for this project that produced output (`exit_status == 0` or absent), fall inside the look-back window (`DELEGATE_SWEEP_WINDOW_HOURS`, default 24, reusing the sweep's env var), are older than a short settle floor (a new `DELEGATE_VERDICT_SETTLE_MIN`, default 3 minutes, so the just-completed turn's delegation is not swept before its artifact has landed), and carry no feedback row referencing their `ts`. This is the same untracked-set query `verdict-sweep.sh` already runs — the design reuses that jq rather than reinventing it.

When the set is empty the hook exits 0 on the cheap path, exactly like the boundary hook's non-boundary commands, so the common case adds no friction. When the set is non-empty it emits `additionalContext` (in warn mode, the default) listing each untracked delegation by `ts`, `recipe`, and `tier`, with an instruction the agent can act on: for each, decide from session memory whether the delegated output was used or rewritten, and record it with `delegate-feedback.sh --ts <ts> --source agent hit|miss`. The agent records the genuine usage verdicts and stops. Mode is env-controlled on the same `warn | off` axis as the boundary hook (there is no `enforce` here — blocking a stop to force verdict recording would be hostile; the value is in the prompt, not in coercion).

Re-prompting is handled by the existing idempotency: once the agent records a verdict, that `ts` has a feedback row and drops out of the next scan. The only re-surface risk is a `ts` the agent declines to judge, which is acceptable — it simply reappears on the next Stop and can be judged then, or aged out of the window. Because the hook fires per-turn, the settle floor plus the empty-set cheap path keep it quiet: it only speaks when there is genuinely an un-judged, settled delegation, and it goes silent again the moment the agent answers.

## Recorder change

`delegate-feedback.sh` gains one optional flag, `--source human|agent`, defaulting to `human` so every existing caller (the maintainer's manual `delegate-feedback.sh hit`, and `verdict-sweep.sh`) is unchanged. When `--source agent` is passed, the emitted feedback row carries `verdict_source:"agent"`. This is the only write-path change; the Stop hook and any future automatic recorder go through the same validated `--ts` path that the sweep already uses, inheriting its `--ts` validation, its row-write shape, and its OTel feedback-span emission. No new row writer is introduced.

## Reporting changes

Two consumers learn the tier distinction. `metrics-summary.sh` and `experiments/quality-trend.py` already build a `ref_ts -> kept` map to compute hit-rate; the change is to partition that map by `verdict_source` so the hit-rate numerator and denominator count human rows only, while coverage counts both. The visible effect is the split shown in the decision preview: coverage rises with agent verdicts included, and a separate line reports the agent tier's volume and its usage rate (how often the agent shipped the delegated output unchanged) as its own figure, clearly labelled so it is never read as a quality number. The lifetime and per-recipe rollups gain an "agent-observed" column alongside the human hit/miss/untracked columns rather than merging the counts.

This also sharpens, rather than muddies, the untracked diagnostic: a delegation is now untracked only if neither the human nor the agent has judged it, which is the number Phase E actually cares about driving under 20%.

## Test and eval impact

The new hook needs its own unit-test file (`tests/test-delegate-verdict-stop-hook.sh`) covering the cheap empty-set exit, the settle-floor exclusion, the window exclusion, the already-tracked exclusion, the per-project scoping, and the warn/off mode gate — the same shape as `tests/test-delegate-boundary-hook.sh`. `tests/test-delegate-feedback.sh` gains cases for the `--source` flag (default human, explicit agent, the row field present/absent). `tests/test-metrics-summary.sh` and `tests/test-quality-trend.sh` gain cases proving the human hit-rate excludes agent rows while coverage includes them — the direct regression guard for the honesty property this design turns on. The trigger-accuracy eval (`eval-skill-triggers.sh`) is untouched: this design changes no `SKILL.md` frontmatter, so the description gate does not apply.

## Honesty safeguards

Three properties keep the agent tier from drifting into the quality signal. The default `--source human` means an accidental omission records as human, which is the conservative direction only if the recorder is the maintainer — so the Stop hook must always pass `--source agent` explicitly, and a test asserts the hook's emitted command contains it. The reporting partition is the load-bearing guard: even if an agent verdict were mis-tagged, the human hit-rate is computed by positive selection of `verdict_source == "human"` (with the legacy no-field rows mapped to human), not by excluding agent — a safer default for the pre-existing corpus. And the agent's instruction text frames the judgment as "did you use it," not "was it good," so the recorded fact matches the tier's stated meaning.

## Rollout

The work decomposes into four reviewable pieces that can land in sequence: the recorder `--source` flag with its tests (smallest, no behaviour change to existing callers); the reporting partition in `metrics-summary.sh` and `quality-trend.py` with its tests (makes the tier visible before anything writes to it); the Stop hook script with its tests and a `docs/boundary-hook.md` sibling section; and finally the global-settings wire-up (a `Stop` entry alongside the existing `PreToolUse` boundary-hook entry) plus the ROADMAP Phase E note recording the mechanism shipped. An ADR (`docs/adr/0015-agent-observed-verdict-tier.md`) should capture why agent verdicts are a separate tier and why the Stop hook is the verdict source, since both are load-bearing and non-obvious. After it ships, the Phase E loop returns to measurement: confirm the untracked rate actually falls as the agent tier fills in, and watch that the human hit-rate stays flat (a drop would mean the partition leaked).
