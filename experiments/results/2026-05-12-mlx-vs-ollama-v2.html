MLX vs Ollama v2

Apples-to-apples re-measurement after PR #114 — same Qwen3.6-35B-A3B 8-bit weights, both backends on API + reasoning-off, Ollama daemon quit during the MLX run. 2026-05-12.
Ollama --ollama-api MLX /v1/chat/completions v2 supersedes v1 for the cross-backend headline
The v1 "15× faster" headline was a methodology artefact. v1 measured Ollama T1–T3 through the ollama run CLI with reasoning on; that streamed every reasoning token through stdout and dominated the wall time. scripts/delegate.sh in production has always used the API path. v2 puts both backends on the same API + reasoning-off regime — the real cross-backend gap is 2×, not 15×.

Opportunities this v2 surfaces

  1. Make MLX the default backend on Apple Silicon. 2× faster + 25% lighter on the same weights, with all the wire-fix work landed in PRs #105/#106/#108/#112/#114. Concrete proposal: add a DELEGATE_BACKEND=auto mode to pick-model.sh — picks MLX if mlx_lm.server is reachable on MLX_HOST, else falls back to Ollama. One env var to opt out.
  2. Multi-tier-resident model serving. The host has 91.5 GB headroom while MLX holds the 35B prose model. That's room for a second MLX-resident model — e.g. a 4-bit Q4 122B premium-general (~60 GB) alongside the 35B prose primary. Would need a small fork-or-wrapper for mlx_lm.server (it serves one model at a time today) or a per-tier server process. Newly feasible with this much headroom.
  3. Promote --ollama-api from runner-flag to baseline default. 8× wall-time saving on Ollama T1–T3 with no accuracy loss. The CLI path should be opt-in (--ollama-cli for back-compat with 2026-04-28 and 2026-05-01 baselines); the API path should be default for new baselines.
  4. Document idle-MLX shutdown in docs/install-mlx.md. The MLX server keeps 36 GB of weights resident between requests; for occasional use, kicking the server between sessions reclaims that memory. The current docs only cover startup; an idle/shutdown section is missing.
  5. Explore Ollama-style restraint as a prompt pattern. Ollama with think:false voluntarily emitted only 1 T3 claim/rep where the prompt allowed up to 4 — and was 100% supported. A future eval could measure whether explicit "prefer fewer claims you're confident about" prompting generalises this restraint to other open-ended tasks (e.g. release-note bullets).

Read this before quoting numbers

v1 inflated the cross-backend gap. The 15× headline was specific to the runner's back-compat CLI regime for Ollama T1–T3, not the production delegate.sh path. Real users were never paying that 519 s wall — they were paying ~64 s, the v2 number.

Stopping the Ollama daemon doesn't make MLX faster. The intuition was reasonable in theory (an idle daemon holding model state would compete) but ollama stop already empties VRAM and the daemon's idle footprint is sub-300 MB. 32 s vs 34 s is noise.

The 91.5 GB of headroom is the most under-utilised resource here. The M5 Max / 128 GB host runs the 35B prose model at 36.5 GB and has room for a second resident model entirely. That's the multi-tier-resident opportunity above — newly feasible because MLX's weights-plus-cache footprint is meaningfully lower than Ollama's.
