# Baseline — 2026-06-03 (MLX backend, all three routed tiers)

First baseline to cover every routed MLX tier in one pass. The `2026-05-27-mlx-baseline.md` run measured only the code and reasoning models and skipped `mlx-community/Qwen3.6-35B-A3B-8bit`, which `pick-model.sh` now routes for **both** the prose and long-context tiers — so the highest-volume model had no recent empirical number. This baseline closes that gap and re-measures the other two for drift. All three models were served via `mlx_lm.server` on `:8080`, 3 reps per task, scored by the deterministic T3/T4/T5/T6 scorers. Backend is MLX because `DELEGATE_BACKEND=auto` resolves to the MLX server on this workstation; the ollama-named command in earlier staleness issues no longer matches the routed path.

## Findings

The prose/long-context model — the data point the previous baseline was missing — is healthy. `Qwen3.6-35B-A3B-8bit` posts perfect structural scores on T5 JSON and T6 regex and a clean T4, with T3 at 0.75: it emits four claims per rep and anchors three of them, which is the same MLX base-prompt citation behaviour measured in PR #115's v2 restraint probe (citation-anchoring is a partial prompt property, not a model defect). Nothing here indicates a routing change for prose or long-context.

The code and reasoning models reproduce the `2026-05-27` numbers exactly — Qwen3-Coder at T3 1.00 / T4 0.83 / T5 1.00 / T6 1.00, and DeepSeek-R1 at T3 0.00 / T4 0.83 / T5 1.00 / T6 0.00 — so there is zero measurable calibration drift over the seven-day window despite recipe and fixture edits in between. DeepSeek-R1's T3 0.00 is the `NONE` restraint posture (zero claims emitted, not hallucination) and its T6 0.00 is a scorer-side artefact where the reasoning model's chain-of-thought wrapping trips the single-line regex check; both are correct-specialist-placement signals seen in the prior baseline, not regressions.

The most notable cross-model result is T4. All three models fail exactly one check per rep with zero stdev, and the failing check is `BODY_NO_PADDING` in every case. A uniform, model-independent failure across three very different architectures confirms this is a prompt-recipe / fixture property — the commit-message padding ceiling documented for the `commit-message` recipe — rather than any one model's limitation. It is the single most worthwhile T4 lever to revisit, and it will not move by swapping models.

All three models are correctly placed in their tiers. No routing changes are indicated.

## Methodology note

This file is named `2026-06-03-baseline.md` (date immediately followed by `-baseline.md`, backend recorded in the body) so the fortnightly `baseline-staleness` routine's date pattern detects it. The `2026-05-27-mlx-baseline.md` infix did not match that pattern, which is why the routine reported false staleness (issue #244) against the older `2026-05-01` ollama baseline even though a newer MLX baseline existed. See ROADMAP "Phase 6 — Recurring maintenance" for the canonical command and the naming requirement.
