===== EVIDENCE =====
Release 2.3 of the metrics service moves the six dashboard panels from range
queries to instant queries. The change touches only the panel query type: it
does not modify any of the underlying query expressions, and it does not add
or remove a datasource. The migration script renames the legacy table
panel_cache to panel_state and backfills the last thirty days of rows so the
new panels have history to draw on from the first render.

The cache key builder was hardened at the same time. It now escapes the
wildcard pattern `cache:*:v2` and the index range `keys[0]` before they
reach the store, so a stray glob can no longer match unrelated entries during
a bulk eviction. The escaping is applied uniformly across the read and write
paths, and the existing key format is otherwise unchanged.

Rollout happened in two waves. The team migrated 40 panels in the first wave
and 55 panels in the second wave, pausing a day between the two to watch the
error budget. The post-migration audit recorded 90 panels as fully verified
against their pre-migration snapshots; the remainder were flagged for a manual
spot check that is still in progress at the time of writing.

The instant-query switch removes the step-interpolation that previously
inflated panel sums on wide time ranges. Early dashboards render correctly in
staging, and the on-call engineer confirmed the panel totals now match the
raw counts pulled directly from the warehouse. A separate operational note:
the on-call rotation for the metrics service is documented in the team
runbook, and the pager schedule was updated in March to add weekend coverage.

Configuration is unchanged for existing deployments. Operators who pinned the
old panel cache TTL via the environment do not need to touch anything; the
default TTL is read from the same place it always was, and the rollback
procedure is the same documented two-step revert.

The dashboards themselves were not restyled in this release. Panel titles,
colour thresholds, and the legend layout are all carried over verbatim from
2.2, and the only visible difference an end user should notice is that the
numbers on wide time ranges are no longer inflated. The grafana provisioning
files were regenerated from the same template, so a diff of the provisioning
directory shows only the query-type field flipping from range to instant on
each of the six panels and nothing else.

Observability for the migration itself was wired through the existing metrics
pipeline rather than a new one. Each wave emitted a counter for panels
migrated and a histogram for per-panel migration latency, both tagged with the
wave number, and those series were watched on a temporary side dashboard that
will be torn down once the manual spot check is signed off. The migration was
watched by hand during the two rollout windows rather than gated on an
automated check.

One correction from the earlier draft of these notes: contrary to the initial
plan, the legacy panel_cache table is NOT dropped in this release. It is
retained read-only for one more cycle so the rollback path stays open, and a
follow-up release will remove it once the manual spot check completes. The
rename to panel_state therefore creates a copy; it does not delete the source.
