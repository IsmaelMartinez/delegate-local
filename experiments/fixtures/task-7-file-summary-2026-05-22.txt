# ADR-0007 — Adopt CloudWAN over Transit Gateway for global routing

## Status

Accepted (2026-04-15).

## Context

Our existing network design uses one AWS Transit Gateway (TGW) per region with cross-region peering attachments configured manually. As we add more regions and more account environments (dev / staging / prod / sandbox), the manual route-table maintenance on every TGW attachment has become a per-change overhead and a frequent source of misconfiguration incidents (4 outages in the last 6 months traced to manual route-table drift).

CloudWAN was launched in 2026 and offers a global routing model where intent is declared at the policy layer (which segments can talk to which other segments) and the underlying attachments and route tables are derived automatically.

## Decision

Adopt AWS CloudWAN as the new global routing layer. Migrate existing TGW attachments onto CloudWAN core network edges region-by-region over Q3 2026.

## Consequences

Positive: route-table drift incidents drop to zero (verified in the pilot region over 8 weeks). Operational overhead per new account drops from ~2 hours of manual route-table edits to ~10 minutes of CloudWAN policy review.

Negative: ~30% higher per-GB data-transfer cost on cross-region links. We accept this trade-off because the operational cost dominated the cloud cost in the incident-analysis below.

Neutral: ingress/egress to on-prem still goes through Direct Connect; CloudWAN does not change that part of the stack.

## Alternatives considered

1. Stay on TGW and invest in automation (CDK constructs that generate route tables). Rejected because the automation surface is still per-region and the cross-region peering remains a manual seam.
2. Use a third-party SD-WAN overlay. Rejected because it adds a vendor dependency for a problem AWS now solves natively.
